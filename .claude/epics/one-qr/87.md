# Task: Click / scan tracking endpoint with opportunistic customer linkage

## Description

`POST /api/qr/event/` accepts `{ slug, action_id?, kind: 'scan'|'click' }` and writes a `QREvent`. If the request carries a `pd_customer` cookie (set elsewhere after a prior booking) the event is linked to that customer and `reward_points` are credited.

## Acceptance Criteria

- [ ] Endpoint accepts both `scan` (no action_id) and `click` (action_id required).
- [ ] Anonymous events never 4xx — missing cookie just stores `customer_id = NULL`.
- [ ] When the cookie resolves to a real `Customer`, the event row is linked and (if `kind == 'click'`) the action's `reward_points` is added to `Customer.points` (column added inline if not present yet).
- [ ] Endpoint accepts CORS from `/qr/[slug]` and from `/admin/qr` preview.
- [ ] 5+ tests: scan anon, click anon, click with cookie + valid customer, click with cookie + bad customer (graceful fallback), scan with invalid slug → 404.

## Technical Details

- `backend/api/qr_public.py`.
- Throttled with DRF's `AnonRateThrottle` at 60/min to deter abuse.
- The `Customer.points` column is added in this task if `retention` hasn't already shipped it; otherwise reuses.
