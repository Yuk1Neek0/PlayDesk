# Task: Customer portal frontend

## Description

The Next.js portal surface at `/s/[slug]/account`. Server component checks the session cookie and renders either the inline login flow (phone → code) or the 4-tab dashboard (Upcoming, History, Loyalty, Profile). Adds `customerFetch` as the customer-side analogue of v6's `adminFetch`.

## Acceptance Criteria

- [ ] `frontend/src/lib/customer-fetch.ts`:
   - Wraps `fetch`, auto-includes `credentials: "include"`, auto-sends `X-PD-Store-Slug` from URL or context.
   - On 401: redirect to `/s/<slug>/account` (the page handles the login UI).
   - Does NOT share cookies / headers with `adminFetch` — different cookie names, different storage.
- [ ] `frontend/src/app/s/[slug]/account/page.tsx` (server component):
   - Reads `pd_customer_session` cookie via `cookies()`.
   - If valid → render `<AccountDashboard customer={...} />`.
   - If invalid → render `<LoginForm storeSlug={params.slug} />`.
   - Branding via `fetchStoreBrand(params.slug)` (v5 pattern) for the header.
- [ ] `frontend/src/app/s/[slug]/account/LoginForm.tsx` (client):
   - Step 1: phone input. On submit → `POST /api/customer-auth/request-code/`. Shows "Code sent to {phone}".
   - Step 2: 6-digit code input. On submit → `POST /api/customer-auth/verify-code/`. On success → reload the page (server component will see the new cookie and render dashboard).
   - Rate-limit feedback: 429 → "Please wait 60 seconds and try again."
   - "Resend code" link after 60 seconds.
- [ ] `frontend/src/app/s/[slug]/account/AccountDashboard.tsx` (client):
   - Tab strip: Upcoming · History · Loyalty · Profile.
   - **Upcoming tab**: list of bookings from `GET /api/me/bookings/?status=upcoming`. Each row: date+time, resource name, "Reschedule" + "Cancel" buttons. Reschedule opens a date/time picker modal. Cancel opens a confirmation modal showing "Free cancellation up to {N} hours before".
   - **History tab**: paginated list from `GET /api/me/bookings/?status=past`. "Load more" button.
   - **Loyalty tab**: fetches `GET /api/me/membership/` → renders balance, tier badge, perks, progress bar to next tier, redeem list. Each reward has a "Redeem" button → `POST /api/me/redeem/`.
   - **Profile tab**: shows phone (read-only), name (editable), "Log out" button → `POST /api/customer-auth/logout/` → redirect to `/s/<slug>/book`.
   - First-time customers with `name=""` → forced "What should we call you?" modal before any tab content.
- [ ] Reschedule modal: date picker pre-filtered to the same resource's availability (call existing `GET /api/availability/?resource_id=...`).
- [ ] All copy / button labels follow existing booking page voice (concise, action-oriented).
- [ ] Mobile-first: dashboard renders cleanly at 375×667 (iPhone SE).
- [ ] `npm run lint && npm run typecheck && npm test && npm run build` passes.

## Technical Details

- Use `next/navigation`'s `useRouter` + `router.refresh()` after login to re-trigger the server-component fetch.
- For the dashboard tabs, prefer URL state (e.g., `?tab=loyalty`) so the browser back button works.
- The dashboard fetch waterfall: hit `/api/me/` for the user, `/api/me/bookings/?status=upcoming` for the default tab. Don't pre-fetch all 4 tabs; lazy-load on tab activation.
- For the Stripe Elements + payment flow integration in tabs: **out of scope for this task — v9 will hook in.**
- Don't share state with `adminFetch`. Customers in the same browser shouldn't accidentally see admin UI if they share a machine — the namespaces are completely separate.
