# Task: End-to-end test for the portal flow

## Description

Playwright e2e covering the happy path: phone → OTP → dashboard → reschedule → cancel → redeem. Mocks the SMS adapter so the OTP code is captured deterministically in the test.

## Acceptance Criteria

- [ ] `frontend/e2e/customer-portal.e2e.ts` (new):
   - **Test 1 — login flow**: navigate to `/s/playdesk-flagship/account`, see login form. Enter `+15551234567` (must be the phone of a seeded customer with bookings). Capture the OTP code from the test SMS sink (either a backend test endpoint or a side-effect on a known table the test reads via API). Enter code. Lands on dashboard.
   - **Test 2 — reschedule a booking**: from dashboard, click Reschedule on an upcoming booking, pick a new slot 2 hours later, submit, see the booking row updated. Assert the booking row in DB shows the new start_at.
   - **Test 3 — cancel a booking**: click Cancel on an upcoming booking, confirm in modal, see the booking disappear from Upcoming and appear in History with "Cancelled" badge.
   - **Test 4 — redeem a reward**: switch to Loyalty tab, click Redeem on an affordable reward, see balance decrease in UI, refresh and confirm persistence.
   - **Test 5 — logout**: click Logout in Profile tab, lands on `/s/playdesk-flagship/book`. Refresh `/s/playdesk-flagship/account` → see login form again.
- [ ] Seed setup: the seed_data command (or a test fixture) creates a customer with phone `+15551234567` at Flagship, with at least 2 upcoming bookings and a $5 reward they can afford.
- [ ] OTP capture mechanism documented in the test file. Recommended: a `?test_mode=1` flag on the request-code endpoint that returns the code in the response (gated by `DEBUG=True` or env-var to never leak in prod).
- [ ] Test added to CI workflow (or runs as part of the existing Playwright suite — check `frontend/playwright.config.ts` for the pattern).
- [ ] Runs green locally with `npm run test:e2e` and in CI.

## Technical Details

- The `?test_mode=1` flag for OTP capture: simplest is `return Response({"request_id": ..., "code": code} if settings.DEBUG and request.query_params.get("test_mode") else {"request_id": ...})`. Acceptable because gated on DEBUG, which is False in prod.
- Alternatively, read the `CustomerOTP` row directly via a test-only API endpoint. Worse — needs more plumbing.
- Use the existing Playwright pattern from `frontend/e2e/multi-location.e2e.ts` for store-context setup.
- Don't run this test in parallel with other e2e tests against the same customer — OTP rate limits.
