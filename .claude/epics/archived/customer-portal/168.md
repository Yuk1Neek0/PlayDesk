# Task: Customer-side loyalty endpoints

## Description

Two endpoints — `/api/me/membership/` and `/api/me/redeem/` — that wrap v4's existing `MembershipView` + `RedeemView` logic with ownership enforcement: instead of taking a customer id from the URL, they read `request.customer`. This keeps v4's atomic redeem logic as the single source of truth.

## Acceptance Criteria

- [ ] `GET /api/me/membership/`:
   - 401 if `request.customer is None`.
   - Returns the exact same payload shape as v4's `MembershipView.get(request, pk)` but for `request.customer`.
   - Implementation: import the helper functions from `core.memberships` and call `_customer_for_request` equivalent, but with `request.customer` instead of the pk parameter.
- [ ] `POST /api/me/redeem/` body `{reward_id}`:
   - 401 if not logged in.
   - Validates the reward belongs to `request.customer.store_id` (else 400).
   - Uses the same `transaction.atomic + select_for_update + balance check + award_points(-cost) + Redemption.create` flow as v4's `RedeemView`.
   - 409 with `{error: "insufficient_points", balance, cost}` if not enough.
   - 201 with `{redemption_id, transaction_id, balance}` on success.
- [ ] Refactor opportunity (optional but encouraged): extract a `core.memberships.compute_membership_payload(customer)` and `core.memberships.redeem(customer, reward, staff)` so both v4's admin endpoints and v7's customer endpoints call the same function. If you take this refactor, ensure v4's existing tests still pass.
- [ ] Tests in `tests/test_me_membership.py`:
   - 401 without cookie
   - membership payload matches v4 admin's payload byte-for-byte (same customer)
   - redeem happy path → balance decreases, Redemption row created, PointTransaction row created
   - redeem cross-store reward (Flagship customer trying to redeem North reward) → 400
   - redeem insufficient balance → 409
   - concurrent redeem race → only one succeeds (uses the existing atomic test pattern)
- [ ] Existing v4 membership tests pass unchanged.

## Technical Details

- The path of least resistance: keep v4's view classes untouched, write `views_me.py::MyMembershipView` that internally calls `MembershipView().get(request, customer.pk)` with a synthetic pk. Or, cleaner: extract the payload-building into a function.
- The redeem flow has a `Customer.objects.select_for_update().filter(pk=customer.pk).first()` lock — preserve it. This is what serializes concurrent redeems.
- `request.user` in v4's `RedeemView` is used as `staff` (the author of the PointTransaction). In v7's customer-initiated redeem, `staff = None` because the customer is doing it themselves. Verify `award_points(author=None)` is supported (the existing v4 code shows it is — `author = request.user if request.user.is_authenticated else None`).
