# Task: Entry-point wiring

## Description

Surface the portal where customers actually live: the QR landing page and the booking page. Each gets a "My Account" link that points to `/s/[slug]/account`. Small task — three files, ~20 lines each.

## Acceptance Criteria

- [ ] `frontend/src/app/qr/[slug]/page.tsx` (v3 one-qr landing):
   - "My Account" button rendered in the top-right header area, next to the existing chat link.
   - Link target: `/s/{slug}/account`.
   - Same visual weight as the chat link.
- [ ] `frontend/src/app/s/[slug]/book/BookingPage.tsx` (v6 booking page):
   - "My Account" header link, top-right.
   - Link target: `/s/{slug}/account`.
- [ ] Optional but encouraged: if the booking-success state (after a customer just made a booking) can detect a fresh `pd_customer` cookie, show a one-line nudge: "Want to manage your bookings? [Create an account]" → links to `/s/{slug}/account`. Account creation is implicit via OTP — same UX.
- [ ] Visual check: take a screenshot at 375×667 of each surface showing the new link.
- [ ] `npm run lint && npm run typecheck && npm test && npm run build` clean.

## Technical Details

- The /qr/[slug] page currently uses cached fetch — be careful not to break it. Just add a link.
- The booking page is client-side beyond the SSR shell; adding a header link should be in the SSR portion, not gated on client state.
- Use Next.js `<Link href={`/s/${slug}/account`}>` for client-side nav.

## Dependencies

- Task 169: the `/s/[slug]/account` route must exist to link to.
