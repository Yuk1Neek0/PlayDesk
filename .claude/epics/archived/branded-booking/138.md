# Task: /api/public/store-brand/ endpoint + accent validation + tests

## Description

Small public endpoint returning the default store's brand fields (name + optional logo_url + optional accent). Validates the accent value against a tiny CSS-grammar regex so an admin typo can't break the page. Cached for 60 seconds.

## Acceptance Criteria
- [ ] `backend/api/views_public.py` (new file) exports `StoreBrandView(APIView)` exposing `GET /api/public/store-brand/`.
- [ ] Response shape: `{ "name": str, "logo_url": str | null, "accent": str | null }`.
- [ ] No authentication required (it's a public branding signal — same posture as `/qr/[slug]` which any anonymous visitor can see).
- [ ] `Store.brand` is read once via `Store.objects.first()` (matches the project's existing single-store convention).
- [ ] Accent validation: regex match against `^(oklch\([^)]+\)|#[0-9A-Fa-f]{6}|rgb\([^)]+\))$`. Non-matching values return `null` in the response (not the malformed string).
- [ ] Cache header `Cache-Control: public, max-age=60` on responses.
- [ ] URL registered in `backend/api/urls.py`.
- [ ] Tests in `tests/test_store_brand_endpoint.py`:
   - Default store with `brand={}` → `{name: "...", logo_url: null, accent: null}`.
   - Store with `brand={logo_url, accent}` (valid) → both surfaced.
   - Store with `brand={accent: "javascript:alert(1)"}` (invalid) → accent is null in response.
   - Anonymous request → 200 (not 401/403).
   - Response has the cache header.

## Technical Details

- The endpoint is intentionally tiny: one DB read, one validation, one serialisation. No DRF serializer class needed — plain dict + `Response()` is fine.
- The accent regex matches the three formats actually documented for the field (oklch / hex / rgb). Add more on demand, not speculatively.
- Single-store assumption: `Store.objects.first()` is the existing pattern (see seed_data + other public views). v6 multi-location replaces this with a routing-driven lookup.
