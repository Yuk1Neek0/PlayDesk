# Task: Booking page consumes helper; renders branded logo + accent

## Description

Wire `fetchStoreBrand()` into the booking page's SSR loader. When `logo_url` is set, render `<Image src={logo_url} ... />` in place of the default SVG logo. When `accent` is set, apply it to the page wrapper via the `--pd-accent` CSS variable so the primary CTA buttons pick it up.

## Acceptance Criteria
- [ ] `frontend/src/app/page.tsx` SSR loader calls `fetchStoreBrand()` in parallel with whatever else it already fetches.
- [ ] When `logo_url` is non-null, the header logo is `<Image src={logo_url} alt={name} width=... height=... />`. When null, the existing default logo SVG renders (unchanged).
- [ ] When `accent` is non-null, the page wrapper element gets `style={{ "--pd-accent": accent } as React.CSSProperties}`. When null, no inline style → the existing default CSS variable wins.
- [ ] The Confirm-booking button (and any other element keyed off `--pd-accent`) reflects the branded colour when `accent` is set.
- [ ] `npm run lint && npm run typecheck && npm run build` clean.
- [ ] A Playwright e2e test asserts: with `brand={}` the page renders the default-state header; with `brand={logo_url, accent}` set on the demo store, the header `<img>` exists with the right `src` and the wrapper has the `--pd-accent` inline style. Lives in `frontend/e2e/branded-booking.e2e.ts`.

## Technical Details

- Use Next's `<Image>` for the logo (project already uses it elsewhere) — but set `unoptimized` if the logo URL is external and not in `next.config.mjs`'s allowed-domains list. Easier: keep an `<img>` tag for v5 to avoid the allowlist dance.
- Do NOT modify the existing `pd-button--primary` CSS — the variable substitution is enough to apply the accent.
- The booking page is in `app/page.tsx` (the root route). Not `app/book/page.tsx` — the booking flow IS the home page.
