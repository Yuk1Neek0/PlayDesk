# Task: BusinessMetricsView endpoint + per-aggregate tests + perf regression

## Description

The composite endpoint backing the dashboard strip. Six aggregates in one transaction over Booking, Customer, OutboundMessage, and QREvent. Returns a fixed-shape payload sized for one round-trip.

## Acceptance Criteria
- [ ] `backend/api/views_metrics.py::BusinessMetricsView(APIView)` exposes `GET /api/admin/metrics/business/?days=N` (default 30).
- [ ] Response shape exactly matches the PRD's six-card structure:
   ```
   { bookings_today, bookings_window, revenue_window, new_customers_window, outbound_24h, qr_window }
   ```
   See `.claude/prds/business-dashboard.md` for field-level types.
- [ ] `bookings_today.count` uses `today_local(store)` for the date filter.
- [ ] `bookings_today.trend_pct_vs_yesterday` is `(today - yesterday) / yesterday * 100`, rounded to 1 decimal; `null` if `yesterday == 0`.
- [ ] `revenue_window.amount_cents` sums `Booking.deposit_amount` for `status='completed'` in the window; returns `0` when Stripe is unconfigured (no `deposit_amount` on rows).
- [ ] `outbound_24h` counts by status (`sent`, `failed`, `queued`) over the last 24 hours.
- [ ] `qr_window.engagement_pct` is `clicks / scans * 100` rounded to 1 decimal; `0.0` when `scans == 0`.
- [ ] Single store scope: uses `Store.objects.first()` (matches existing single-store convention).
- [ ] Response cache header: `Cache-Control: private, max-age=30`.
- [ ] URL registered in `backend/api/urls.py`.
- [ ] Tests in `tests/test_business_metrics_endpoint.py`:
   - Empty DB → all counts/sums are 0, engagement_pct = 0.0.
   - Seeded mixed bookings → `bookings_today` reflects only today's, `bookings_window` reflects the window.
   - Seeded trend test: 2 bookings today, 1 yesterday → trend = +100.0; 1 today, 0 yesterday → trend = null.
   - Stripe-unconfigured (no deposit_amount) → revenue = 0.
   - QR engagement math: 10 scans, 3 clicks → engagement_pct = 30.0.
   - Cache header present.
- [ ] Perf regression test: seed 10 000 bookings + 100 000 QREvents, assert endpoint returns in <300ms p95 over 10 runs.

## Technical Details

- Each metric is a single `aggregate()` call. Avoid loading rows into Python. Use `Count`, `Sum`, `F` expressions.
- One DB transaction wraps the six aggregates so the snapshot is internally consistent.
- The `today_local` helper is called once at the top of the view and reused for both `bookings_today` and the yesterday-trend calculation.
- The perf test can use `pytest-benchmark` if available, otherwise a `time.perf_counter()` measurement.
