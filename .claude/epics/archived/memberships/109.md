# Task: Backfill data migration — seed PointTransaction from Customer.total_visits

## Description

Data migration that gives every existing customer a starting balance equal to `total_visits * store.points_per_booking`. Written as one `source = "backfill"` `PointTransaction` per customer. Idempotent — skipped for any customer that already has any transaction (so re-running the migration is a no-op).

This lets the membership module land without orphaning a customer's pre-v4 visit history.

## Acceptance Criteria
- [ ] Migration file: `backend/core/migrations/000ZZ_backfill_points.py` (number determined by `makemigrations`).
- [ ] For each `Customer` where no `PointTransaction` exists, writes one row with `delta = total_visits * store.points_per_booking`, `source = "backfill"`, `reference = "backfill-v4"`, `balance_after = delta`.
- [ ] Customers with `total_visits = 0` get no row (no zero-delta transactions).
- [ ] The migration calls `award_points()` rather than reimplementing the ledger write — keeps the single-write-path invariant.
- [ ] Reverse migration: deletes `PointTransaction` rows with `source = "backfill"` AND `reference = "backfill-v4"`. Other rows survive.
- [ ] Forward migration is safe to re-run (no duplicate rows on a second `migrate`).
- [ ] Tests: empty-db forward → no-op + no rows; seeded-customers forward → one row each; double-forward → still one row each; reverse → only backfill rows removed; customer with mixed history (one earn + one backfill via second forward) → reverse only drops the backfill.

## Technical Details

- The migration imports `award_points` from `core.memberships` rather than calling `PointTransaction.objects.create()` directly — this avoids divergence from the helper's locking + `balance_after` logic.
- Idempotence check: `PointTransaction.objects.filter(customer=c).exists()` — single boolean query, indexed lookup.
- `Store.points_per_booking` is read once per store inside the loop and cached in a dict to avoid N+1 queries on `c.store.points_per_booking`.
