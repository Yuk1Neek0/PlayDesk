# Task: Customer-facing check-in surface

## Description

The customer's check-in path: SMS link → public page → one-tap → confirmation. Backend assigns the token at booking create, the SMS confirmation includes the URL, the public page shows a friendly card with a check-in button, and the POST flips the booking state idempotently.

## Acceptance Criteria

- [ ] `backend/api/serializers.py::BookingCreateSerializer.create`:
   - Assigns `validated_data["check_in_token"] = generate_unique_check_in_token()` before super().create().
- [ ] `backend/api/serializers.py::BookingSerializer`:
   - Add `check_in_token`, `checked_in_at` to `fields` + `read_only_fields`.
- [ ] `backend/api/views_checkin.py` (new):
   - `class CheckInInfoView(APIView)` — `GET /api/c/<token>/`:
     - `authentication_classes: list = []`, `permission_classes: list = []` (public).
     - Looks up `Booking.objects.filter(check_in_token=token.upper()).first()` — 404 if not found.
     - Returns `{booking_id, status, checked_in_at, customer_name, resource_name, start_time, end_time, store_slug, can_check_in: bool, message: str}`.
     - `can_check_in=True` only if `status == CONFIRMED` and `checked_in_at is None`.
     - `message` field per state:
       - CONFIRMED → `"Ready to check in"`.
       - CHECKED_IN → `"Already checked in at {time}"`.
       - COMPLETED → `"This booking is complete"`.
       - CANCELLED → `"This booking was cancelled"`.
       - PENDING_PAYMENT → `"Please complete your deposit first"`.
   - `class CheckInActionView(APIView)` — `POST /api/c/<token>/check-in/`:
     - Public. Same lookup.
     - If already `CHECKED_IN` → 200 with the existing payload (idempotent, no error).
     - If status is `CONFIRMED` → set `status=CHECKED_IN`, `checked_in_at=timezone.now()`, save, return updated payload.
     - Else → 409 with the `message` from the GET shape.
- [ ] URL wiring in `backend/api/urls.py`: `path("c/<str:token>/", CheckInInfoView.as_view())`, `path("c/<str:token>/check-in/", CheckInActionView.as_view())`.
- [ ] `backend/outbound/templates.py::TEMPLATES["booking_confirmation"]`:
   - Extend the EN + ZH bodies to append the check-in URL. Recommended exact change to the en body:
     - Old: `"Hi {customer_name}, your booking at {store_name} is confirmed for {start_time} on {resource_name}. See you then!"`
     - New: `"Hi {customer_name}, your booking at {store_name} is confirmed for {start_time} on {resource_name}. Check in on arrival: {checkin_url}"`
   - ZH side: equivalent append.
- [ ] Wherever `booking_confirmation` is enqueued (probably `outbound/api.py::enqueue_message` callers or a signal handler), the context dict must now include `checkin_url = f"{settings.SITE_URL}/c/{booking.check_in_token}"`. Audit all enqueue sites that use `template_key="booking_confirmation"`.
- [ ] `frontend/src/app/c/[token]/page.tsx` (new) — server component:
   - SSR-fetches `GET /api/c/<token>/` (use `BACKEND_ORIGIN` env, same pattern as `/qr/[slug]/page.tsx`).
   - On 404 → `notFound()`.
   - Renders centered card with store branding (call `fetchStoreBrand(payload.store_slug)`).
   - First name extraction: `customer_name.split(" ")[0]`.
   - Renders `<CheckInClient info={payload} />` for the interactive bit.
- [ ] `frontend/src/app/c/[token]/CheckInClient.tsx` (new) — client component:
   - If `can_check_in=true`: show "Welcome, {firstName}! Tap to check in" + a big "I'm here" button. On click → POST `/api/c/<token>/check-in/`, show "✓ Welcome — please enjoy your session" on 200.
   - If `can_check_in=false`: render the `message` in a state-appropriate styled card.
   - Always shows the resource + time block for context.
- [ ] Tests in `tests/test_checkin_api.py`:
   - GET on a CONFIRMED booking → 200 + `can_check_in: true`.
   - GET on a CANCELLED booking → 200 + `can_check_in: false` + correct message.
   - GET on unknown token → 404.
   - POST on CONFIRMED → 200, status flipped, checked_in_at populated.
   - POST on already-CHECKED_IN → 200 (idempotent), same payload, no error.
   - POST on CANCELLED → 409.
   - Token is uppercase-normalized: `/c/abc12345/` matches `check_in_token="ABC12345"`.
- [ ] Frontend test `frontend/src/app/c/[token]/page.test.tsx` (vitest, optional but encouraged):
   - Mock the API response, assert the welcome card renders.
- [ ] `npm run lint && npm run typecheck && npm test && npm run build` clean.

## Technical Details

- The `BACKEND_ORIGIN` env var pattern is already used by `/qr/[slug]/page.tsx` for SSR fetches inside the Next.js container — reuse the same `backendOrigin()` helper.
- Use `fetchStoreBrand(payload.store_slug)` to keep the check-in card branded — same pattern as `/qr/[slug]/page.tsx` (after the v6 follow-on fix).
- Don't add a custom auth requirement — token IS the credential. This is the design.
- The SITE_URL setting was added in v9 billing-payments and should be available; check `settings.SITE_URL` exists.
