# Task: Admin check-in visibility + manual override

## Description

Staff need to see who's checked in and override when reality doesn't match: walk-ins lost their SMS, customers tapped the wrong link, etc. This task adds the admin booking-list badge, the booking-detail check-in panel, and two staff endpoints with audit trail via CustomerNote.

## Acceptance Criteria

- [ ] `backend/api/views_checkin.py` (extend from task #197):
   - `class AdminCheckInView(APIView)` — `POST /api/admin/bookings/<int:pk>/check-in/`:
     - No permission_classes (project convention — v10a's StaffOnlyMiddleware gates if landed; otherwise open).
     - Looks up `Booking.objects.filter(pk=pk)` scoped to `request.store` (404 if cross-store).
     - If status is `CONFIRMED` → set `CHECKED_IN`, stamp `checked_in_at`, save. Else 400 with a clear error per current state.
     - Writes `CustomerNote(customer=booking.customer, author=request.user, body=f"Manually checked in by staff at {checked_in_at:%Y-%m-%d %H:%M}.")`.
     - Returns the updated BookingSerializer payload.
   - `class AdminUndoCheckInView(APIView)` — `POST /api/admin/bookings/<int:pk>/undo-check-in/`:
     - If status is `CHECKED_IN`: revert to `CONFIRMED`, set `checked_in_at=None`, save.
     - Writes `CustomerNote(body="Check-in undone by staff.")`.
     - Returns BookingSerializer payload.
     - Else 400.
- [ ] URL wiring: `path("admin/bookings/<int:pk>/check-in/", ...)`, `path("admin/bookings/<int:pk>/undo-check-in/", ...)`.
- [ ] `frontend/src/components/admin/checkin-badge.tsx` (new):
   - Small inline component. Props: `{checkedInAt: string | null, status: string}`.
   - Renders "—" if not checked in.
   - Renders "✓ {time}" with a green dot if `checked_in_at` is set.
   - Renders "✓ Complete" if status is COMPLETED.
- [ ] Admin bookings list (`frontend/src/app/admin/bookings/page.tsx` or wherever):
   - Add a new column "Check-in" rendering `<CheckInBadge ... />`.
   - Filter dropdown: All / Not yet checked in / Checked in.
   - Filter backend: extend the existing admin-bookings list endpoint to accept `?checked_in=yes|no`. If admin-bookings doesn't exist yet as a separate endpoint, add a simple query param to whatever lists bookings for the admin (DRF ViewSet filter or list endpoint).
- [ ] Admin booking detail page (locate first — may be a section inside the customers list, or a dedicated route):
   - Add a "Check-in" panel showing the badge + two buttons "Manual check-in" (only if `status==CONFIRMED`) and "Undo check-in" (only if `status==CHECKED_IN`).
   - Buttons POST to the new endpoints via `adminFetch`.
   - On success → refresh the detail (refetch or revalidate the data).
- [ ] Tests in `tests/test_admin_checkin.py`:
   - Manual check-in CONFIRMED → 200, status flipped, CustomerNote created.
   - Manual check-in already-CHECKED_IN → 400.
   - Undo CHECKED_IN → 200, status reverted, CustomerNote created.
   - Undo non-CHECKED_IN → 400.
   - Cross-store access → 404.
- [ ] Frontend smoke test for the CheckInBadge component (vitest).
- [ ] `npm run lint && npm run typecheck && npm test && npm run build` clean.

## Technical Details

- If task #196's `BookingStatus` choice landed correctly, the queryset filters are straightforward (`Booking.objects.filter(status="checked_in")`).
- For `request.user` attribution on CustomerNote: if v10a hasn't landed, `request.user` could be AnonymousUser. Handle gracefully: `author=request.user if request.user.is_authenticated else None` (mirrors how memberships endpoints already handle this).
- The admin booking-list filter is opt-in via query param; default behavior unchanged.
- The "Undo check-in" button is intentionally not a confirm-modal — undo is reversible by clicking Manual check-in again, so a modal would be friction without value.
