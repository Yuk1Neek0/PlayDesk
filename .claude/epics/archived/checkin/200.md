# Task: End-to-end coverage for check-in

## Description

The Playwright e2e that ties everything together: book a session, retrieve the check-in token (test endpoint or DB), simulate the customer arriving and scanning the link, verify status updates, verify the staff dashboard reflects it.

## Acceptance Criteria

- [ ] `frontend/e2e/checkin.e2e.ts` (new):
   - **Test 1 — customer check-in happy path**: book a slot through the customer booking page → look up the just-created booking's `check_in_token` via a test-only endpoint or via a small Playwright helper that calls the admin API as `playdesk_staff` → navigate to `/c/<token>/` → see the welcome card → click "I'm here" → see "✓ Welcome" → verify admin booking list shows checked-in badge.
   - **Test 2 — re-tap idempotency**: visit `/c/<token>/` again after check-in → see "Already checked in at {time}" message → no duplicate state change.
   - **Test 3 — cancelled booking can't check in**: book → cancel via admin → visit `/c/<token>/` → see "This booking was cancelled" message → check-in button absent.
   - **Test 4 — admin manual check-in for walk-in**: as a staff user, find a confirmed booking in the admin list → click "Manual check-in" → verify badge flips → verify CustomerNote was created with the "Manually checked in by staff" body.
   - **Test 5 — sweeper promotion** (backend, not e2e but in this task): integration test that creates a CHECKED_IN booking with `end_time` 60 min ago, runs the management command via `call_command("auto_complete_checked_in")`, asserts the booking is now COMPLETED and `booking_thank_you` was enqueued.
- [ ] Token-discovery for the e2e: either:
   - (a) Add a Playwright fixture that creates the booking + reads back the token via the admin booking-detail API (preferred — exercises the admin endpoint).
   - (b) Add a `?test_mode=1` flag on booking-create that includes the check-in_token in the response (gated on `DEBUG=True`, mirrors v7's customer-portal test pattern).
- [ ] Verify the existing 687-test backend suite + 102 frontend test suite still pass post-merge.
- [ ] `npm run lint && npm run typecheck && npm test && npm run build` clean.

## Technical Details

- Use the seed_data fixture's seeded customer + bookings as the starting point if a fresh booking is overkill.
- For the staff-side tests, depend on v10a's `playdesk_staff` if landed (the e2e starts by logging in); else use the existing `force_login` or similar pattern that earlier e2e tests use.
