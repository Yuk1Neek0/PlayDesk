# Task: Check-in schema foundation

## Description

The data layer for check-in: two new `Booking` fields, one new `BookingStatus` choice, a unique-token generator, and an idempotent backfill migration for existing rows.

## Acceptance Criteria

- [ ] `backend/core/models.py::Booking`:
   - Add `check_in_token: CharField(max_length=12, unique=True, db_index=True, blank=True)`.
   - Add `checked_in_at: DateTimeField(null=True, blank=True)`.
- [ ] `backend/core/models.py::BookingStatus`:
   - Add `CHECKED_IN = "checked_in", "Checked in"` between `CONFIRMED` and `COMPLETED`.
- [ ] `backend/core/tokens.py` (new):
   - `generate_check_in_token() -> str` — returns an 8-char string from the base32 alphabet minus ambiguous chars (`0`, `O`, `1`, `I`, `l`). Uses `secrets.choice` over the alphabet. Allowed alphabet: `"23456789ABCDEFGHJKLMNPQRSTUVWXYZ"`.
   - `generate_unique_check_in_token(max_retries=5) -> str` — calls `generate_check_in_token` in a loop until `Booking.objects.filter(check_in_token=t).exists()` is False. Raises `RuntimeError` if 5 retries fail (basically never — 32^8 = 1.1e12 keyspace).
- [ ] `backend/core/migrations/000X_booking_check_in_fields.py`:
   - Adds `check_in_token` (nullable initially) + `checked_in_at` + status choice.
   - Adds the unique constraint AFTER backfill (see next migration).
- [ ] `backend/core/migrations/000Y_backfill_check_in_token.py`:
   - `RunPython` that loops over `Booking.objects.filter(check_in_token="")` (or `__isnull=True`) and assigns `generate_unique_check_in_token()` to each.
   - Use `apps.get_model("core", "Booking")` for forward migration safety. For the token generator: import locally or duplicate the generator into the migration (since `core.tokens` may import settings).
   - Reverse: `RunPython.noop`.
- [ ] After the backfill, add the unique constraint on `check_in_token` via a third migration OR use `unique=True` from the start with a sentinel value (e.g., uuid pre-fill, then real tokens) — pick whichever Django's `makemigrations` likes.
- [ ] Tests in `tests/test_check_in_token.py`:
   - `generate_check_in_token()` returns 8 chars from the allowed alphabet.
   - 1000 calls → all unique (statistical, fine to be a smoke test).
   - `generate_unique_check_in_token` retries on collision (mock `Booking.objects.filter().exists()` to return True once).
   - After migration, every existing seeded booking has a non-empty `check_in_token`.
- [ ] `python manage.py makemigrations --check` clean (no drift). Add explicit `name=` on any new index.
- [ ] Existing 687-test backend suite passes.

## Technical Details

- The alphabet excludes `0/O/1/I/l` so customers reading the URL off a printed sign don't typo. 32 chars × 8 positions = 1.1 × 10^12 keyspace — well past any brute force concern at PlayDesk request volume.
- `secrets.choice` (not `random.choice`) — these are URL tokens, treat as auth credentials even though they're public-by-design.
- Don't worry about case-insensitivity in the alphabet — base32 is uppercase-only by convention; the URL handler can `.upper()` the inbound token before lookup.
- The migration's local generator avoids the `RuntimeWarning: Accessing model X in migration` lecture. If the agent prefers to import from `core.tokens` and it works in CI, that's fine.
