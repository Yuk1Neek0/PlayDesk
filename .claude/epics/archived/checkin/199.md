# Task: Auto-completion sweeper

## Description

Closes the booking lifecycle loop. A management command run hourly via cron promotes `CHECKED_IN` bookings to `COMPLETED` after their end_time + 30-minute grace, enqueuing the existing `booking_thank_you` SMS. Idempotent + safe to re-run.

## Acceptance Criteria

- [ ] `backend/core/management/commands/auto_complete_checked_in.py` (new):
   - `Command.handle(**options)`:
     - Computes `cutoff = timezone.now() - timedelta(minutes=30)`.
     - Queries `Booking.objects.filter(status="checked_in", end_time__lt=cutoff)`.
     - For each, in a single transaction:
       - Sets `status = "completed"`.
       - Calls `enqueue_message(booking.customer, "booking_thank_you", {customer_name: ..., store_name: ...})` via the v4 outbound API.
       - Saves.
     - Logs count of promoted rows.
   - Options:
     - `--grace-minutes N` — overrides the 30-min default.
     - `--dry-run` — logs what would happen without writes.
     - `--store SLUG` — scope to one store (useful for staging).
- [ ] Idempotent: re-running immediately is a no-op (all eligible rows already promoted).
- [ ] Safe under no-rows: completes cleanly when zero bookings match (no errors, no SMS sent).
- [ ] Tests in `tests/test_auto_complete_sweeper.py`:
   - One CHECKED_IN booking with `end_time = now() - 60min` → promoted to COMPLETED, SMS enqueued (mock outbound).
   - One CHECKED_IN booking with `end_time = now() - 10min` (under grace) → NOT promoted.
   - One CONFIRMED booking (no check-in) → NOT promoted.
   - Dry-run mode → no DB writes, no SMS, but logs match.
   - Re-running after a promotion → zero new promotions.
   - Filter by --store=slug → only that store's bookings affected.
- [ ] Document the cron pattern in the command's docstring:
   ```
   Suggested cron (hourly):
     0 * * * * cd /app && python manage.py auto_complete_checked_in
   ```
- [ ] `python manage.py auto_complete_checked_in` runnable from the docker compose backend container.
- [ ] Existing 687-test backend suite passes.

## Technical Details

- Use `Booking.objects.filter(...).select_for_update()` inside a `transaction.atomic()` to avoid double-promoting a row that's being concurrently modified (e.g. staff undoing a check-in at the moment the cron runs).
- The grace period of 30 min is a reasonable compromise: a 2-hour booking that runs 15 min over still completes correctly; an actively-extending session doesn't get prematurely closed. Configurable for flexibility.
- The SMS enqueue is best-effort: if the outbound queue is down, the booking should still be promoted (the SMS is a notification, not state). Use try/except around the enqueue with a log line on failure.
- For testing time-sensitive behavior, use `freezegun` or the existing project pattern (check `tests/test_outbound_sender.py` for the project's freezing approach).
