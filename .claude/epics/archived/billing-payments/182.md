# Task: Deposit charge at booking creation

## Description

Wire the booking-create flow to create a Stripe PaymentIntent when a deposit is required, and render Stripe Elements on the booking page so the customer can pay inline.

## Acceptance Criteria

- [ ] Booking-create endpoint logic:
   - Compute `deposit = calc_deposit(store, resource, total_amount)` (task 180 helper). `total_amount` comes from v8 if landed, else `resource.hourly_rate * hours`.
   - If `deposit == 0`:
     - Create booking with `payment_status='not_required'`, `status='confirmed'`. Return as today.
   - If `deposit > 0`:
     - Create booking with `payment_status='pending_payment'`, `status='pending'`, `deposit_amount=deposit`.
     - Create PaymentIntent: `stripe.PaymentIntent.create(amount=int(deposit*100), currency=store.currency, transfer_data={"destination": store.stripe_account_id}, metadata={"booking_id": booking.id})`.
     - Save `booking.payment_intent_id = intent.id`. Create a Payment row with `kind="deposit"`, `status="pending"`, `stripe_payment_intent_id=intent.id`.
     - Return `{booking_id, client_secret: intent.client_secret, requires_payment: True, deposit_amount: deposit}`.
- [ ] Frontend booking page:
   - On booking-form submit, hit the endpoint. If `requires_payment: True`:
     - Render Stripe Elements (`@stripe/react-stripe-js`) with `clientSecret`.
     - Customer enters card → call `stripe.confirmPayment({elements, confirmParams: {return_url: ...}})`.
     - On success: redirect to a confirmation page; the actual `payment_status` flip happens via webhook (task 183) asynchronously.
     - Show "Confirming payment, please wait…" until either webhook arrives (polling `GET /api/me/bookings/{id}/payment-status/` every 2s, max 30s) or timeout → "Contact staff".
- [ ] New endpoint `GET /api/bookings/{id}/payment-status/` (public, no auth, just for the post-payment poll):
   - Returns `{payment_status, status}`. Safe to expose because the booking_id is opaque + only useful for status read.
- [ ] `frontend/src/lib/stripe.ts`:
   - Loads `loadStripe(STRIPE_PUBLISHABLE_KEY)` lazily.
- [ ] New frontend deps: `@stripe/stripe-js` + `@stripe/react-stripe-js`.
- [ ] Tests in `billing/tests/test_booking_payment_flow.py`:
   - Booking with `deposit_mode='none'` → no Stripe call, status confirmed (mocked stripe to assert no calls).
   - Booking with `deposit_mode='percentage' deposit_value=30` on $80 total → PaymentIntent created with amount=2400 (cents), booking status=pending, payment_status=pending_payment.
   - Booking with resource override → override wins.
   - Payment-status endpoint returns current state.
- [ ] Existing booking tests pass (they all use `deposit_mode='none'` by default).

## Technical Details

- Stripe amounts are integer cents. `int(decimal_amount * 100)`. Be careful with rounding — quantize the decimal first.
- `transfer_data.destination` requires Stripe Connect Standard accounts to be `charges_enabled=True`. If not, the PaymentIntent.create call will fail — handle by returning 503 "Store is not yet ready to accept payments" + log.
- The `confirmParams.return_url` is needed for redirect-style payment methods (e.g., 3DS auth). For card-only flows, this URL is the booking-confirmation page.
- The polling for payment-status is a fallback; the webhook is the source of truth. The poll just bridges UX between card-confirmed and webhook-arrived.
- **Composition with v8**: if v8's task 176 has landed, `total_amount` is already on the booking row before deposit calc. If not, this task should add a stub computation. Use the `Booking.balance_amount` property helper from task 181 — it handles both cases.
