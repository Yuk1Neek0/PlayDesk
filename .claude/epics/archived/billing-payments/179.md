# Task: Stripe Connect onboarding

## Description

Per-store Stripe Connect Standard onboarding. Adds Stripe SDK + new fields on Store + onboarding endpoint + return-URL handler + admin settings page. Foundation for every other v9 task.

## Acceptance Criteria

- [ ] New pip dep `stripe` added to `backend/requirements.txt` (or pyproject.toml).
- [ ] New env vars in `backend/config/settings.py`:
   - `STRIPE_SECRET_KEY` (required if `STRIPE_TEST_MODE` is False; sk_test_ key fine in test mode).
   - `STRIPE_PUBLISHABLE_KEY` (pk_*).
   - `STRIPE_WEBHOOK_SECRET` (whsec_*).
   - `STRIPE_TEST_MODE` (default True in dev).
- [ ] New Django app `billing` added to `INSTALLED_APPS`.
- [ ] `billing.stripe_client.get_stripe()` returns a configured stripe module instance (keyed off settings).
- [ ] Store augmentation in `core/models.py`:
   - `stripe_account_id: CharField(max_length=64, null=True, blank=True)`
   - `stripe_charges_enabled: BooleanField(default=False)`
   - `currency: CharField(max_length=3, default="USD")`
- [ ] Migration `core/migrations/000N_store_stripe_fields.py`.
- [ ] `POST /api/admin/stripe/connect/` (store-scoped via `request.store`):
   - If `store.stripe_account_id` is null → create Stripe Connect Standard account via `stripe.Account.create(type="standard", country=..., capabilities={...})`. Persist id.
   - Generate onboarding link via `stripe.AccountLink.create(account=..., refresh_url=..., return_url=..., type="account_onboarding")`.
   - Return `{onboarding_url, account_id}`.
- [ ] `GET /api/admin/stripe/return/` (return URL after onboarding):
   - Reads `account_id` from query.
   - Calls `stripe.Account.retrieve(...)` → updates `store.stripe_charges_enabled` from `account.charges_enabled`.
   - Redirects to `/admin/settings/payments/`.
- [ ] `frontend/src/app/admin/settings/payments/page.tsx`:
   - Shows: connection status badge ("Connected · acct_xxxx" or "Not connected"), `charges_enabled` indicator, "Connect" / "Reconnect" button.
   - On connect click → `POST /api/admin/stripe/connect/` → window.location to the returned onboarding_url.
- [ ] Tests in `billing/tests/test_stripe_connect.py`:
   - Connect endpoint creates account when none exists (mock `stripe.Account.create`).
   - Connect endpoint reuses existing account_id.
   - Return endpoint updates `charges_enabled` (mock `stripe.Account.retrieve`).
   - 400 if `STRIPE_SECRET_KEY` is missing AND `STRIPE_TEST_MODE` is False.
- [ ] If real Stripe keys provided in `.env`, manual smoke: connect → onboard → return → see badge updated. Document in PR.
- [ ] Existing backend test suite passes.

## Technical Details

- Use `stripe-mock` Docker image for integration tests if `STRIPE_TEST_MODE=True` AND `stripe-mock` is reachable. Otherwise mock at the module level with `unittest.mock.patch("stripe.Account.create")`.
- `STRIPE_TEST_MODE=True` AND `STRIPE_SECRET_KEY` unset → degraded mode: endpoints log "Stripe not configured, skipping" and return 503. Don't crash.
- The onboarding flow opens Stripe's hosted page; we never see card details.
- `refresh_url` and `return_url` are built from `settings.SITE_URL` or `request.build_absolute_uri()`. Use `request.build_absolute_uri('/admin/settings/payments/refresh')` for the refresh URL.
