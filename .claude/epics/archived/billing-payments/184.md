# Task: Refunds + balance charge + orphan sweep

## Description

Three admin-side flows that close the payment loop: cancel triggers auto-refund per matrix, staff can charge remaining balance via a Stripe Checkout link delivered over SMS, and a cron sweeps orphaned `pending_payment` bookings.

## Acceptance Criteria

- [ ] Cancel endpoint (both admin-side AND `/api/me/bookings/{id}/cancel/` from v7 if landed):
   - If `booking.payment_status in ['deposit_paid','paid_in_full']`:
     - `refund_amount = calc_refund_amount(store, booking)` (task 180).
     - If `refund_amount > 0` → `stripe.Refund.create(payment_intent=booking.payment_intent_id, amount=int(refund_amount*100))`.
     - Webhook handler `charge.refunded` (task 183) will flip booking.payment_status.
     - Write a temporary Payment(kind='refund', status='pending', amount=-refund_amount) so the admin sees the pending refund immediately.
   - Send SMS via v4 outbound: template `booking_refunded` with the refund amount.
- [ ] `POST /api/admin/bookings/{id}/charge-balance/`:
   - Computes `balance = booking.balance_amount` (the property from task 181).
   - If `balance <= 0` → 400 "Already paid in full".
   - Creates a Stripe Checkout Session: `stripe.checkout.Session.create(mode="payment", line_items=[{...}], payment_intent_data={"transfer_data": {"destination": store.stripe_account_id}}, metadata={"booking_id": booking.id}, success_url=..., cancel_url=...)`.
   - Creates a Payment row with `kind="balance"`, `status="pending"`.
   - Sends an SMS to the customer via v4 outbound (template `balance_charge_link`) with the Checkout URL.
   - Returns `{checkout_url, balance_amount}` to the staff UI.
- [ ] `python manage.py sweep_pending_payments`:
   - Finds bookings with `payment_status='pending_payment' AND created_at < now() - 10min`.
   - Cancels each: sets `status='cancelled'`, `payment_status='not_required'` (the PaymentIntent will auto-expire or can be explicitly cancelled via `stripe.PaymentIntent.cancel`).
   - Logs each sweep.
   - Idempotent: runs are safe to repeat.
- [ ] Outbound templates added:
   - `booking_refunded`: `"PlayDesk: $${amount} refunded to your card. Refund typically arrives in 5-10 business days."`
   - `balance_charge_link`: `"PlayDesk: complete your booking payment ($${balance}): {checkout_url}"`
- [ ] Tests in `billing/tests/test_refund_and_balance.py`:
   - Cancel within 100% refund window → stripe.Refund.create called with full deposit (mocked).
   - Cancel within 50% window → half refunded.
   - Cancel within 0% window → no Refund.create call.
   - Charge-balance creates Checkout Session + Payment row + sends SMS.
   - Charge-balance on already-paid booking → 400.
   - Sweep finds 10-min-old pending bookings + cancels them.
   - Sweep does NOT touch bookings < 10 min old.
- [ ] Cron registration: document how to run the sweep (e.g., add to docker-compose as a separate hourly job, or document the cron line for production). For v9, a `Makefile` target `make sweep` is acceptable.
- [ ] Existing tests pass.

## Technical Details

- The cancel flow's "temporary pending refund Payment row" is for UI immediacy — the webhook will create the canonical refund row later. Distinguish via `stripe_event_id` being null on the temporary one. Webhook handler should UPDATE the row (matching by `stripe_charge_id` + `kind='refund'`) rather than creating a duplicate.
- Stripe Checkout Session URLs expire in 24 hours by default. Document this in the customer SMS template if you want.
- The sweep handles the edge case where the PaymentIntent was created but the customer abandoned the page — without the sweep, the booking sits in pending_payment forever blocking the resource slot (the GIST exclusion constraint counts pending bookings).
- Actually wait — does the GIST exclusion include pending bookings? Check the model. If yes, that's the right behavior (block the slot to prevent double-booking during the 10-min window). If no, sweep is less urgent but still good hygiene.
