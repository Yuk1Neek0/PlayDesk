# Task: Staff auth backend endpoints

## Description

Three new endpoints under `/api/staff/` that wrap Django's built-in `authenticate` + `login` + `logout` + session machinery. Plus a `playdesk_staff` seeded user so developers can log in immediately after pulling the branch.

## Acceptance Criteria

- [ ] `backend/api/views_staff_auth.py` (new):
   - `StaffLoginView` (POST `/api/staff/login/`):
      - Body: `{username, password}`.
      - Calls `authenticate(request, username=..., password=...)`. If None → 401 `{detail: "Invalid credentials."}`.
      - If user found but `not user.is_staff` → 403 `{detail: "Not a staff account."}`.
      - On success: `login(request, user)` (writes the session), returns 200 `{id, username, is_superuser, is_staff}`.
      - Rate-limited: 5 attempts per username per 15 min via `django.core.cache.cache.add(key, 1, timeout=900)`. On limit → 429.
   - `StaffLogoutView` (POST `/api/staff/logout/`):
      - Calls `logout(request)`. Always 200 `{ok: true}`.
   - `StaffMeView` (GET `/api/staff/me/`):
      - If `request.user.is_authenticated AND request.user.is_staff` → 200 `{id, username, is_superuser, is_staff}`.
      - Else 401 `{detail: "Not authenticated."}`.
- [ ] All three views are CSRF-exempt (`@csrf_exempt`) so the frontend can call them as JSON without a token round-trip. Acceptable because the login/logout endpoints don't mutate authenticated state (login mutates anonymous state; logout is harmless to spam). The risk surface is the same as any public form.
- [ ] Wire URLs in `backend/api/urls.py`: `path("staff/login/", StaffLoginView.as_view())`, etc.
- [ ] `backend/config/settings.py`:
   - `SESSION_COOKIE_AGE = 14 * 24 * 3600`  # 14 days
   - `SESSION_COOKIE_SAMESITE = "Lax"`
   - `SESSION_COOKIE_HTTPONLY = True`
   - `LOGIN_URL = "/staff/login/"`
   - `SESSION_COOKIE_SECURE = env.bool("SESSION_COOKIE_SECURE", default=False)`  # True in prod
- [ ] `backend/core/management/commands/seed_data.py`:
   - Idempotent block: `user, _ = User.objects.get_or_create(username="playdesk_staff", defaults={...})`.
   - On create: set `user.is_staff = True`, `user.is_active = True`, `user.set_password("playdesk_staff_demo_pw")`, `user.save()`.
   - At command end, print `"Demo staff: username=playdesk_staff password=playdesk_staff_demo_pw"`.
- [ ] Tests in `tests/test_staff_auth.py`:
   - Login with valid creds → 200, session created, /api/staff/me/ subsequently returns the user.
   - Login with wrong password → 401, no session.
   - Login with non-staff user → 403.
   - Logout → /api/staff/me/ subsequently returns 401.
   - Rate limit: 6th login attempt within 15 min → 429.
   - /api/staff/me/ anonymous → 401.
- [ ] Existing 687-test backend suite passes (no admin-endpoint changes yet — that's task #191).

## Technical Details

- Use Django's `User = get_user_model()` to support a future custom user model swap.
- For rate-limit key: `f"staff_login_attempts:{username}"`. Use `cache.get_or_set(key, 0)` + `cache.incr(key)` pattern. The "5 attempts per 15 min" window resets when a successful login happens (clear the key on 200).
- The seeded password is intentionally weak — it's a dev fixture, not production. The `seed_data.py` print line is the security disclosure.
- Don't import `request.user` lazily — Django's middleware sets it.
