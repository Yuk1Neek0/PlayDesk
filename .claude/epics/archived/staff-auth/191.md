# Task: Admin API gate + test migration

## Description

The load-bearing security change: every URL matching `/api/admin/*` now requires a Django session with `is_staff=True`. Reverses the project's prior "no API-level permission gates" convention for this URL prefix only — customer endpoints stay open. Existing tests get a `force_login_staff` pytest fixture so they can drive admin endpoints without copy-pasting the login dance.

## Acceptance Criteria

- [ ] `backend/core/middleware.py::StaffOnlyMiddleware`:
   - Process every request; if `request.path.startswith("/api/admin/")`:
     - Not authenticated → return `JsonResponse({"detail": "Authentication required."}, status=401)`.
     - Authenticated but `not request.user.is_staff` → `JsonResponse({"detail": "Staff access required."}, status=403)`.
     - Else: pass through.
   - Other paths: pass through unchanged.
   - Registered in `MIDDLEWARE` AFTER `AuthenticationMiddleware` (so `request.user` exists) and AFTER `CurrentStoreMiddleware` (so admin endpoints still see `request.store`).
   - Module docstring explicitly notes: "This middleware reverses the historical 'no API-level permission gates' convention for /api/admin/* only — see PRD .claude/prds/staff-auth.md."
- [ ] `backend/conftest.py` — add the fixture:
   ```python
   @pytest.fixture
   def staff_user(db):
       from django.contrib.auth import get_user_model
       User = get_user_model()
       u, _ = User.objects.get_or_create(
           username="test_staff", defaults={"is_staff": True}
       )
       u.set_password("test_staff_pw"); u.save()
       return u

   @pytest.fixture
   def staff_client(client, staff_user):
       client.force_login(staff_user)
       return client
   ```
- [ ] Update every existing test that hits `/api/admin/*` to use `staff_client` instead of `client`. Mechanical sweep — find with `grep -rl "client.get.*\"/api/admin/\|client.post.*\"/api/admin/" backend/tests/`. Expected ~30 files affected (the admin-customer tests, admin-bookings tests, memberships admin tests, outbound admin tests, dashboard tests, etc.).
- [ ] Tests in `tests/test_staff_only_middleware.py`:
   - Anonymous GET `/api/admin/customers/` → 401.
   - Non-staff user (logged in as a regular User) → 403.
   - Staff user → 200.
   - `/api/customers/` (no admin prefix) → 200 anonymous (not gated).
   - `/api/me/` (customer-portal) → 401 (gated by v7's CustomerSessionMiddleware, not StaffOnlyMiddleware) — confirms the two middlewares don't interfere.
- [ ] Full backend suite passes (the fixture migration should account for every previously-open admin test).
- [ ] CSRF: verify the admin frontend's `adminFetch` includes `X-CSRFToken` header (Django reads it for unsafe methods). If missing → add. This is in v6's `adminFetch`; check the existing implementation.

## Technical Details

- The middleware short-circuits BEFORE the view runs, returning a JsonResponse. No view code change needed.
- `client.force_login(user)` (Django test client) is the cleanest path — bypasses the rate limiter, no password dance.
- The fixture creates a single shared `test_staff` user per test. If a test needs a *specific* staff user (e.g. for CustomerNote attribution assertions), it can build its own.
- One subtle gotcha: the existing `tests/test_admin_*.py` test files may create their own User instances and `force_login` them. Those tests already work; just verify they also have `is_staff=True` set. If a test created a user without `is_staff`, the middleware will return 403 — that's a real test bug (the test was always wrong; it just got away with it because there was no gate).
