# Task: Frontend session provider + login page

## Description

The frontend session-detection surface. New `<StaffSessionProvider>` mounted at the admin layout root that calls `/api/staff/me/` on mount; 401 → redirect to `/staff/login`. New `/staff/login/` page with a real form that POSTs to `/api/staff/login/`. Extend `adminFetch` to trigger logout on 401 mid-session.

## Acceptance Criteria

- [ ] `frontend/src/lib/staff-session.tsx` (new):
   - `StaffSessionContext` provides `{user, ready, error, logout}`.
   - `<StaffSessionProvider>` on mount: `fetch("/api/staff/me/", {credentials: "include"})`. On 200 → `setUser(payload)`, `setReady(true)`. On 401 → `setUser(null), setReady(true)`, and if `pathname.startsWith("/admin/")` and `pathname !== "/staff/login"` → `router.replace("/staff/login?next=" + encodeURIComponent(pathname))`.
   - `useStaffSession()` hook returns the context value.
   - `logout()` action: POSTs `/api/staff/logout/`, clears local state, `router.push("/staff/login")`.
- [ ] `frontend/src/app/staff/login/page.tsx` (new) — server component:
   - Reads the `sessionid` cookie. If present, calls `/api/staff/me/` from the SSR side; if 200 → `redirect("/admin")`.
   - Else renders `<LoginForm next={searchParams.next || "/admin"} />` (a client component).
- [ ] `frontend/src/app/staff/login/LoginForm.tsx` (new) — client component:
   - Form: username + password fields + submit button.
   - Submit: `POST /api/staff/login/` with JSON body. On 200 → `router.push(next)`. On 401 → inline error "Invalid credentials". On 429 → "Too many attempts, try again in a few minutes." On 403 → "This account isn't a staff account."
- [ ] `frontend/src/lib/admin-fetch.ts`:
   - On any response with status 401 → trigger the logout flow (probably best as a callback registered by `<StaffSessionProvider>`, or a simple `window.location.assign("/staff/login?next=" + currentPath)` as a fallback).
- [ ] `frontend/src/app/admin/layout.tsx`:
   - Wrap children in `<StaffSessionProvider>`. Keep the existing `<StoreProvider>` from v6 outside or inside — doesn't matter; both should be present.
   - Until `ready` is true, render a small spinner (don't flash unauthenticated state).
- [ ] `frontend/src/app/staff/layout.tsx` (new, minimal):
   - Just `{children}` plus the existing site shell. The login page shouldn't be wrapped in StaffSessionProvider (it'd cause infinite redirect loops if 401 fires from the provider).
- [ ] Tests in `lib/staff-session.test.tsx` (vitest):
   - Provider mounts → calls `/api/staff/me/` (mocked) → on 200, exposes user.
   - On 401 with pathname starting `/admin/` → redirect called.
   - On 401 with pathname `/staff/login` → no redirect (would loop).
   - logout() POSTs and clears state.
- [ ] `npm run lint && npm run typecheck && npm test && npm run build` clean.

## Technical Details

- `credentials: "include"` is non-optional on every fetch — sessions are cookie-based.
- Use `next/navigation`'s `useRouter` + `usePathname` for the redirect logic. The provider is a client component (`"use client"`).
- Don't call `/api/staff/me/` in a useEffect that re-runs on every route change — that'd thrash the endpoint. Call once on provider mount; the provider lives at the admin layout root and doesn't unmount during admin navigation.
- The SSR-side check in `/staff/login/page.tsx` is an optimization, not a correctness requirement. If the cookie's invalid the redirect just skips. Falls back to client-side flow gracefully.
