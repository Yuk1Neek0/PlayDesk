# Task: Admin endpoints + customer-detail "Outbound messages" section

## Description

The user-visible surface for the slice. Two list endpoints (per-customer log and failure inspection) and a new "Outbound messages" card on the existing `/admin/customers/[id]` detail page.

No standalone admin page in v4 — everything is per-customer.

## Acceptance Criteria
- [ ] `GET /api/admin/outbound/?customer_id=N&limit=20` returns the last N messages for that customer, newest-first, fields: `{ id, template_key, body, status, scheduled_for, sent_at, failure_reason, channel }`.
- [ ] `GET /api/admin/outbound/?status=failed&limit=50` returns recent failures across all customers (admin-wide failure inspection), with customer name / phone joined in for triage.
- [ ] Both endpoints behind the admin auth gate.
- [ ] Both endpoints store-scoped to the request user's accessible stores (no cross-store leak).
- [ ] `frontend/src/app/admin/customers/[id]/page.tsx` extended with a new `<OutboundMessagesSection />` card.
- [ ] The card lists the last 20 messages with: a status pill (queued / sent / failed / cancelled in distinct colours), timestamp (relative + absolute on hover), body preview (truncate >120 chars), failure reason on hover for failed rows.
- [ ] Empty state: "No outbound messages yet." Not an error.
- [ ] `npm run lint && npm run typecheck && npm run build` clean.

## Technical Details

- File: `backend/api/views_outbound.py` (new) + URL conf addition.
- Backend list endpoint is one DRF `ListAPIView` parameterised by query string; serializer is lightweight.
- Frontend file: `frontend/src/components/admin/outbound-messages-section.tsx`. Fetches once on mount via the existing API client; polls every 30s while the page is open so a freshly-sent message shows up without a manual reload.
- Reuse existing `pd-pill`, `pd-card` design tokens.
- The failure-inspection endpoint is a list-only convenience for ops; no frontend page consumes it in v4 (used via curl / `gh` for now).
