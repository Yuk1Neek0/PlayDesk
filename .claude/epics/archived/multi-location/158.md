# Task: CurrentStoreMiddleware + request.store resolver + tests

## Description

The foundation under everything else in this epic. One Django middleware that resolves `request.store` from header / cookie / URL kwarg / fallback. Once this lands, every view / template / agent loop can read `request.store` instead of `Store.objects.first()`. No behaviour change in this task — the fallback resolves to the same single store the app uses today.

## Acceptance Criteria
- [ ] `backend/core/middleware.py::CurrentStoreMiddleware` exists, sets `request.store: Store` on every request.
- [ ] Resolution order (first hit wins): URL kwarg `store_slug` → `X-PD-Store-Slug` request header → `pd_store_slug` cookie → alphabetically-first `Store` (fallback).
- [ ] If the resolver finds a store via header but not cookie, the response sets the `pd_store_slug` cookie to that slug (so cross-tab nav keeps the same store). `SameSite=Lax`, no `Secure` flag (the admin runs over HTTP in dev too).
- [ ] If the resolved slug doesn't match any store, falls through to the next resolver step (don't 404).
- [ ] Registered in `backend/config/settings.py::MIDDLEWARE` after `AuthenticationMiddleware`.
- [ ] Tests in `tests/test_current_store_middleware.py`:
   - Header set → `request.store.slug == header value`.
   - Cookie set, no header → `request.store.slug == cookie value`.
   - URL kwarg set, no header / cookie → `request.store.slug == URL value`.
   - Nothing set → falls back to alphabetically-first store.
   - Header set + cookie set → header wins.
   - Invalid slug in header → falls through to cookie.
   - Header-driven resolution sets the cookie in the response.
   - All resolutions cache the `Store` instance — no per-view DB query for it.
- [ ] Existing 485-test backend suite passes unchanged (the fallback keeps single-store behaviour identical to today).

## Technical Details

- The middleware runs once per request; cache the resolved `Store` on `request._cached_store` so repeated `request.store` access doesn't re-query.
- `Store` lookup by slug uses `Store.objects.filter(slug=...).first()` (returns None on no-match → fall through). One query per request.
- The fallback (`Store.objects.order_by("slug").first()`) is also one query; cache it at module level if you're worried about per-request cost (today there's one store, so it's a single 1-row lookup).
- Module-level cache invalidation: a Store create/delete is rare and admin-driven; an O(1) cost is fine without invalidation in v6.
- The middleware sees ALL requests including static files and health checks; gate the resolution on `request.path.startswith("/api/")` if perf matters.
