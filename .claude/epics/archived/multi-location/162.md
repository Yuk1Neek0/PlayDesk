# Task: Customer-facing store-scoped URL prefix

## Description

Adds `/s/[slug]/book` as the new customer-facing booking URL, mirroring today's `/`. Old `/` becomes a 302 redirect to `/s/<default-store-slug>/book` so existing bookmarks / QR cards keep working. The booking flow's API calls include the store slug so the backend resolver scopes everything correctly.

## Acceptance Criteria
- [ ] `frontend/src/app/s/[slug]/book/page.tsx` (new) — SSR loader that:
   - Resolves `params.slug` (the URL store slug).
   - Calls `fetchStoreBrand(params.slug)` (helper signature evolves to accept an optional slug — see Technical Details).
   - Renders the booking page client component (renamed from today's `BookingPage`).
- [ ] `frontend/src/app/s/[slug]/book/BookingPage.tsx` (new) — client component, near-identical to today's `BookingPage`. The only difference: every API call (availability, booking-create) passes the `X-PD-Store-Slug` header so the backend resolver picks the URL store.
- [ ] `frontend/src/app/page.tsx` — replaced with a server-side 302 redirect to `/s/<default-store-slug>/book`. The default store comes from a small new endpoint `GET /api/public/default-store/` returning `{slug}` (single-row read, cached).
- [ ] `backend/api/views_public.py` — extend with `DefaultStoreView` returning `{slug}` of the alphabetically-first store. No auth; 60s cache header.
- [ ] `backend/api/urls.py` — register `GET /api/public/default-store/`.
- [ ] `frontend/src/lib/store-brand.ts::fetchStoreBrand(slug?: string)` — extends the v5 signature with an optional slug. When set, the endpoint URL becomes `/api/public/store-brand/?store=<slug>`; when omitted, the URL stays `/api/public/store-brand/` (default-store fallback).
- [ ] `backend/api/views_public.py::StoreBrandView` — reads `?store=<slug>` query param. When present, returns that store's brand. When absent, returns `request.store.brand` (which the middleware resolves to the alphabetically-first store via fallback).
- [ ] Existing Playwright tests pass:
   - `booking.e2e.ts` follows the `/` → `/s/playdesk-flagship/book` redirect transparently and still passes.
   - `branded-booking.e2e.ts` works against the new URL.
   - `admin.e2e.ts` unchanged.
- [ ] New Playwright test `frontend/e2e/store-routing.e2e.ts`:
   - Direct nav to `/s/playdesk-flagship/book` renders the flagship-branded booking page.
   - Direct nav to `/` 302-redirects to the default store's `/s/.../book` URL.
   - Booking made via the URL is associated with the URL's store (verified by switching to that store in admin and finding the row).
- [ ] `npm run lint && npm run typecheck && npm test && npm run build` clean.

## Technical Details

- The 302 redirect is a server-side redirect using Next.js's `redirect()` from `next/navigation` in a server component. Avoid client-side redirects (they break direct-bookmark navigation).
- The default-store endpoint is a tiny separate endpoint instead of inlining the lookup into the SSR loader, so it benefits from HTTP-level caching and can be reused by other surfaces (e.g. customer portal in v7).
- `fetchStoreBrand` adds the optional slug parameter; existing callers in the v5 `/qr/[slug]` route don't need changes (they don't pass the slug because the QR page already has a URL slug — they could be migrated to pass it explicitly in a follow-up, but not in v6).
- The booking-flow's API calls already use the project's `fetch` shape. Wrap them via a small helper that injects `X-PD-Store-Slug: params.slug` for all calls on this route. Don't reuse `adminFetch` from #004 — that one reads from `StoreContext` (admin-only). This is a customer-facing variant: `customerFetch(slug, input, init)`.
- The `/` route's redirect target must come from `/api/public/default-store/` (not hardcoded), so a multi-store deployment that drops the flagship works correctly.
