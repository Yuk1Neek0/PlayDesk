# Task: Send pipeline â€” send_campaign runner + state machine + recipient snapshot

## Description

The synchronous runner that ties everything together. Flips status, snapshots recipients into `CampaignRun` rows in one transaction, then iterates the runs calling `send_campaign_message`. Enforces the campaign state machine â€” once `sent`, no edit / no re-send.

## Acceptance Criteria
- [ ] `backend/campaigns/runner.py::send_campaign(campaign_id, sent_by) -> dict` returns `{ sent: N, failed: N, skipped: N, snapshot_count: N }`.
- [ ] Behaviour:
  - Loads the campaign with `select_for_update()`.
  - If `status != "draft"` and `status != "scheduled"`: raises `CampaignAlreadyProcessed` (mapped to 409 by the view).
  - Flips `status` to `"sending"`, sets `sent_by`.
  - Inside one `transaction.atomic()`:
    - Calls `customers_for(campaign.segment)` to get the recipient queryset.
    - For each customer: if `"sms_opt_out" in customer.tags`, creates a `CampaignRun(status="skipped_optout")`. Otherwise creates a `CampaignRun(status="queued")`.
    - Writes `campaign.recipient_snapshot_count = total_runs`.
    - Commits.
  - Iterates the `queued` runs:
    - Renders the body via `rendering.SafeFormatter` against `{"customer": customer, "store": campaign.store}`.
    - Calls `send_campaign_message(customer, body, reference=f"campaign:{campaign.id}:run:{run.id}")`.
    - On `ok=True`: `run.status='sent'`, `sent_at=now()`, `outbound_message_id` set if returned.
    - On `ok=False`: `run.status='failed'`, `failure_reason=reason or "unknown"`.
  - After the loop: flips `campaign.status` to `"sent"`, sets `sent_at`.
- [ ] Recipient cap: if `customers_for(segment).count() > 1000`, raises `CampaignTooLarge` (mapped to 400) â€” campaign is NOT touched (still draft).
- [ ] `cancel_campaign(campaign_id)`: allowed only in `draft` or `scheduled`; flips to `cancelled`; no runs written.
- [ ] Tests:
  - Happy path: 5 customers in segment, 1 opted-out â†’ snapshot=5, 4 sent, 1 skipped_optout.
  - Re-send protection: second `send_campaign` call on a sent campaign raises `CampaignAlreadyProcessed`.
  - Edit protection (in the API view tests, not here): PATCH on sent campaign returns 409. (Logic-side: ensure `send_campaign` is the only writer that flips status.)
  - Recipient cap: 1001-customer segment raises `CampaignTooLarge`, campaign stays `draft`.
  - Cancellation: `cancel_campaign` on draft â†’ cancelled; on sent â†’ raises.
  - Body rendering: missing `{customer.something}` key raises a clear `KeyError` (via `SafeFormatter`).
  - Both send impls covered: monkeypatch the import switch to exercise stub + real paths in the same test file using the `force_send_impl` fixture from task 003.

## Technical Details

- File: `backend/campaigns/runner.py` + `backend/campaigns/rendering.py` (SafeFormatter, same pattern as outbound's templates).
- The two-phase commit pattern (snapshot in one transaction, sends outside) is deliberate â€” keeping the send loop inside a single long transaction would hold locks for seconds.
- The state machine is enforced inside `send_campaign` itself, not at the view layer. The view layer translates the exception to an HTTP code.
- `select_for_update()` on the campaign row prevents two simultaneous `send_campaign` calls on the same id.
