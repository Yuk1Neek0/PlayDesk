# Task: Rotating-key data layer + minter

## Description

The foundational schema + rotation engine. One new model in a new `checkin` Django app, one Store field, a management command + cron pattern that mints new keys when the prior one ages past the store's configured interval.

## Acceptance Criteria

- [ ] New Django app `checkin` added to `INSTALLED_APPS`.
- [ ] `checkin.models.RotatingCheckinKey`:
   - `store` (FK Store, related_name="checkin_keys").
   - `key`: `CharField(max_length=16, unique=True, db_index=True)`. Generated from alphabet `"23456789ABCDEFGHJKLMNPQRSTUVWXYZ"` (no ambiguous chars), 10 chars = ~50 bits entropy.
   - `created_at`: `DateTimeField(auto_now_add=True)`.
   - `expires_at`: `DateTimeField` — set to `created_at + store.checkin_rotation_minutes` at mint time.
   - `superseded_at`: `DateTimeField(null=True, blank=True)` — set when the *next* key replaces this one (gives the previous key a grace window).
   - Indexes: `(store, expires_at)`, `(store, superseded_at)`.
- [ ] `core.Store` augmentation:
   - `checkin_rotation_minutes`: `PositiveSmallIntegerField(default=15)`.
- [ ] Migrations:
   - `core/migrations/000X_store_checkin_rotation.py` — additive field.
   - `checkin/migrations/0001_initial.py` — RotatingCheckinKey.
- [ ] `checkin.services` module:
   - `mint_key(store) -> RotatingCheckinKey` — generates a new key, supersedes the previous active one (sets `superseded_at = now()` on it), creates + returns the new row.
   - `get_active_key(store, *, grace_seconds=60) -> RotatingCheckinKey | None` — returns the most recent non-expired key. If both a fresh key AND a superseded-but-recently-superseded key exist, the fresh one wins (admin display + new scans use it).
   - `resolve_key(key_str) -> RotatingCheckinKey | None` — for `/api/c-in/lookup-key/`. Returns the row if:
     - It exists,
     - AND it's not expired (`expires_at > now()`),
     - AND if superseded, the supersession was less than `grace_seconds` ago (so a scan mid-rotation still works).
- [ ] `python manage.py rotate_checkin_keys`:
   - For each store, if `get_active_key(store)` is None OR the most recent key was created more than `store.checkin_rotation_minutes - 1` minutes ago, mint a new key.
   - Options: `--force` (mint regardless of age), `--store SLUG` (one store only), `--dry-run`.
   - Idempotent + safe to re-run.
   - Suggested cron pattern in docstring: `* * * * * python manage.py rotate_checkin_keys` (every minute).
- [ ] Token generator helper (likely in `checkin/services.py` or a tiny `checkin/tokens.py`):
   - `_generate_key()` — `secrets.choice` over the alphabet, 10 chars.
   - Retry on uniqueness collision (essentially never at 32^10 keyspace).
- [ ] Tests in `checkin/tests/test_keys.py`:
   - `mint_key` creates a fresh key + sets `superseded_at` on the previous one (idempotency: re-calling mint doesn't double-mint within rate).
   - `get_active_key` returns the freshest non-expired key.
   - `resolve_key` returns the row when valid; None when expired; None when unknown; the row when superseded < grace_seconds ago.
   - `rotate_checkin_keys` mints when interval exceeded; no-op when fresh; respects `--store` filter.
   - Existing 739-test backend suite passes.

## Technical Details

- The grace window: when the cron fires at minute X+15 and supersedes key A with key B, customers who scanned key A in the last 15 seconds shouldn't get a 410. 60s grace covers them.
- `mint_key` is called inside a transaction so the supersede + create are atomic.
- `Store.checkin_rotation_minutes` defaults to 15. Allowed range enforced by `MinValueValidator(1)` + `MaxValueValidator(60)`.
- The `name=` on each new index should be explicit (avoid the v4 campaigns CI flake).
