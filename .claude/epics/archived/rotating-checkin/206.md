# Task: Customer-facing rotating-checkin UI

## Description

The page customers land on after scanning the in-store QR. Multi-step: phone → OTP → match resolution (0 / 1 / many bookings) → confirmation. Mobile-first; reuses v10b styling.

## Acceptance Criteria

- [ ] `frontend/src/app/c-in/page.tsx` (server component):
   - Reads `?k=<key>` from search params.
   - If missing → render "No code provided — please scan the QR at the door".
   - SSR-fetches `POST /api/c-in/lookup-key/`. On 410 → render expired-card with "Ask staff for the current QR" message.
   - On 200 → render `<RotatingCheckinClient storeName={...} storeSlug={...} keyValue={...} />`.
   - Uses `fetchStoreBrand(storeSlug)` for header branding.
- [ ] `frontend/src/app/c-in/RotatingCheckinClient.tsx` (client component) — state machine:
   - **Step 1 (phone)**: phone input (international format hint) + Continue → `POST /api/c-in/request-otp/`. Show "Code sent to {phone}".
   - **Step 2 (OTP)**: 6-digit code input + Verify → `POST /api/c-in/verify-and-find/`.
     - On 401 → "Wrong code, try again" + stay on step 2.
     - On 200 with empty matches → step 3a (walk-in message).
     - On 200 with 1 match → step 3b (single-booking confirm).
     - On 200 with 2+ matches → step 3c (disambiguation).
   - **Step 3a (walk-in)**: "No bookings found on this phone today — please see staff to register a walk-in session." Plus a "Start over" link.
   - **Step 3b (single match)**: "Welcome back, {first_name}! Ready to check in for {resource_name} at {time}?" + "I'm here" button → `POST /api/c-in/check-in/` → step 4.
   - **Step 3c (multiple matches)**: list of bookings as cards with "Check in" button per row. Each click → `check-in` → step 4 for that booking. The customer can repeat for additional bookings.
   - **Step 4 (success)**: "✓ Checked in — enjoy your session" + (if multi-booking flow) "Or check in another booking" returns to step 3c.
- [ ] Use `customerFetch` pattern (no auth header, includes credentials, store slug from URL kwarg if needed).
- [ ] Render all error states inline; never silent. Network errors → "Connection problem — please try again."
- [ ] Mobile-first: works at 375×667 (iPhone SE).
- [ ] Tests in `frontend/src/app/c-in/RotatingCheckinClient.test.tsx`:
   - Phone-entry submit → mocked request-otp call → step transitions to OTP entry.
   - OTP submit with 401 → renders "Wrong code" inline.
   - verify-and-find returns empty matches → step 3a walk-in card renders.
   - verify-and-find returns 2 matches → step 3c disambiguation list renders.
- [ ] `npm run lint && npm run typecheck && npm test && npm run build` clean.

## Technical Details

- The `BACKEND_ORIGIN` env var pattern for SSR fetches matches v6 + v10b `/qr/[slug]/page.tsx`.
- For the OTP input, the 6-digit auto-advance pattern from v7's customer-portal LoginForm.tsx is a fine reference.
- First-name extraction for the welcome message: `customer_name.split(" ")[0]` falls back to "there" if blank.
- For the walk-in card, copy should be friendly + direct: don't make customers feel like they failed.
