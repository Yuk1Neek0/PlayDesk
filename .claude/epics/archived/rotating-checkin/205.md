# Task: Public check-in endpoints with OTP

## Description

Four public endpoints that together drive the rotating-checkin flow. Reuses v7's `CustomerOTP` for the SMS verification — no parallel OTP infra.

## Acceptance Criteria

- [ ] `backend/checkin/views.py` (new):
   - All four views have `authentication_classes: list = []`, `permission_classes: list = []`.
   - **`POST /api/c-in/lookup-key/`** body `{key}`:
     - Calls `checkin.services.resolve_key(key)`.
     - Returns 200 `{store_slug, store_name, expires_at}` if valid.
     - Returns 410 `{detail: "Code expired — please ask staff for the current QR."}` if expired/unknown.
   - **`POST /api/c-in/request-otp/`** body `{key, phone}`:
     - Validates key first (resolves it; 410 if invalid).
     - Normalises `phone` to E.164 (use existing helper).
     - Calls v7's `request_code` endpoint INTERNALLY (or factor out the helper function in views_customer_auth.py so both endpoints share it). Don't HTTP-call v7; share the function.
     - Returns 200 `{request_id}` on success; 429 on rate limit.
   - **`POST /api/c-in/verify-and-find/`** body `{key, phone, code}`:
     - Validates key.
     - Verifies the OTP via v7's `verify_code` flow (shared helper). Failure → 401.
     - On success: looks up bookings for that phone in the resolved store within the same-day ±2hr window:
       ```python
       from datetime import timedelta
       now = timezone.now()
       window_start = now - timedelta(hours=2)
       window_end = now + timedelta(hours=2)
       bookings = Booking.objects.filter(
           resource__store=store,
           customer_phone=phone_normalized,
           status=BookingStatus.CONFIRMED,
           start_time__gte=window_start,
           start_time__lte=window_end,
       ).select_related("resource").order_by("start_time")
       ```
     - Returns `{bookings: [{id, resource_name, start_time, end_time, can_check_in}, ...]}`. Empty list = walk-in path.
     - **Important**: do NOT consume the OTP here. The customer may need to call `check-in` next with the same code (typical UX: verify-and-find returns matches, customer picks one, check-in flips). Make OTP single-use by storing a "verified for this phone" flag with short TTL (5 min) — use `cache.set(f"checkin_verified:{phone}", True, timeout=300)`.
   - **`POST /api/c-in/check-in/`** body `{key, phone, booking_id}`:
     - Validates key.
     - Checks `cache.get(f"checkin_verified:{phone}") is True`. If not → 401 "Verification expired, please re-enter your code."
     - Looks up the booking by `id` AND phone AND store AND within the window. 404 if no match.
     - If already `CHECKED_IN` → 200 idempotent (return same payload).
     - If `CONFIRMED` → flip to `CHECKED_IN`, stamp `checked_in_at = now()`, save.
     - Returns booking payload.
     - On success: delete the cache key (single-use verification).
- [ ] URL wiring in `backend/checkin/urls.py` + include under `/api/c-in/` in `backend/config/urls.py`.
- [ ] Tests in `checkin/tests/test_views.py`:
   - lookup-key happy + expired + unknown.
   - request-otp triggers v7 SMS send (mock outbound), persists OTP row, returns 200.
   - verify-and-find with valid OTP returns matches; sets `checkin_verified` cache; doesn't consume OTP yet.
   - verify-and-find with invalid OTP returns 401.
   - check-in with valid verification flips status; deletes cache; idempotent.
   - check-in without verification cache returns 401.
   - check-in for a booking outside the time window returns 404.
   - Cross-store booking lookup returns empty list (not 404) — the search just doesn't match.
- [ ] Existing 739-test backend suite passes.

## Technical Details

- The OTP sharing: factor `views_customer_auth._do_request_code(phone, store_slug, channel="sms")` and `_do_verify_code(phone, code, store_slug)` out as helpers in `core/customer_auth.py` (or wherever), call them from both v7's views AND v11a's views. Less code duplication than HTTP-self-call.
- The verification cache key: `f"checkin_verified:{phone}"` — by phone NOT by OTP, because the customer should be able to type the OTP once and have it cover both verify-and-find AND check-in.
- The "no consume on verify-and-find, consume on check-in" pattern means the OTP row is still active until check-in succeeds. If the customer abandons the flow, the OTP just expires naturally (v7's 10-min TTL).
- `customer_phone` on Booking: use the normalised phone, matching how v7 stores it.
