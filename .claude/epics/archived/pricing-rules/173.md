# Task: Pricing strategies + registry

## Description

The model + strategy framework that all subsequent v8 tasks plug into. New `pricing` Django app with a single model, an abstract base class for rule evaluation, 5 concrete strategy classes, and a registry dict.

## Acceptance Criteria

- [ ] New Django app `pricing` added to `INSTALLED_APPS`.
- [ ] `pricing.models.PricingRule`:
   - `store` (FK Store, on_delete=CASCADE), `name`, `description` (blank ok), `enabled` (default True), `priority` (PositiveSmallIntegerField, default 100), `stackable` (default True), `rule_type` (CharField choices), `params` (JSONField), `applies_to_resource` (FK Resource, nullable, on_delete=CASCADE), `created_at`, `updated_at`.
   - `rule_type` choices: `peak_hours`, `day_of_week`, `member_tier`, `min_duration`, `bracket_rate`.
   - Meta: ordered by `priority, id`. Indexes on `(store, enabled)` and `(store, rule_type)`.
- [ ] `pricing.strategies.base.RuleStrategy` (ABC):
   - `params_schema(cls) -> dict` — JSON schema for the params field (used for admin form generation + server-side validation).
   - `validate_params(cls, params: dict) -> None` — raises `ValidationError` on bad shape.
   - `applies(self, rule: PricingRule, ctx: QuoteContext) -> bool`
   - `compute(self, rule: PricingRule, ctx: QuoteContext, running_total: Decimal) -> Decimal` — returns a signed adjustment.
- [ ] `pricing.strategies.QuoteContext` dataclass:
   - `resource: Resource`, `start_at: datetime`, `end_at: datetime`, `customer: Optional[Customer]`, `customer_tier: Optional[RewardTier]`, `hours: Decimal`, `base_amount: Decimal`.
- [ ] Five concrete strategies, one file each under `pricing/strategies/`:
   - `peak_hours.py`: params `{days: ["mon","tue",...], start_hour: int, end_hour: int, adjustment_pct: int}`. Applies if `ctx.start_at.weekday()` is in `days` AND `start_at.hour >= start_hour AND < end_hour`. Returns `running_total * (adjustment_pct / 100)`.
   - `day_of_week.py`: params `{days: [...], flat_rate: Decimal}` OR `{days: [...], discount_pct: int}`. If flat_rate, returns `flat_rate * hours - running_total` (effectively overrides total). If discount_pct, returns `-running_total * (discount_pct / 100)`.
   - `member_tier.py`: params `{tier_id: int, discount_pct: int}`. Applies if `ctx.customer_tier and ctx.customer_tier.id == tier_id`. Returns `-running_total * (discount_pct / 100)`.
   - `min_duration.py`: params `{min_hours: int, discount_pct: int}`. Applies if `ctx.hours >= min_hours`. Returns `-running_total * (discount_pct / 100)`.
   - `bracket_rate.py`: params `{brackets: [{max_hours: int|null, rate: Decimal}]}`. Applies always (when this rule is active for the resource). Returns `sum_over_brackets(min(remaining_hours, bracket_size) * rate) - running_total` — effectively overrides the base total.

## Technical Details

- Use `Decimal` everywhere; never `float`. `from decimal import Decimal`.
- `compute` returns SIGNED adjustments (negative for discounts, positive for surcharges).
- For `bracket_rate` and `day_of_week`-with-flat-rate, the returned adjustment "overrides" the running total by computing `desired_total - running_total`. The engine in task 175 just sums adjustments.
- `params_schema` returns a minimal JSON Schema dict (object with required keys + types) used by both the admin form generator (task 177) and server-side validation. Don't over-engineer — the v8 schemas are simple.
