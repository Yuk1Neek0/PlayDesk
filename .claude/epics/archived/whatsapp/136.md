# Task: pick_channel_for() helper + enqueue_message integration

## Description

Implicit channel-preference resolver. Looks up a customer's most recent inbound conversation; if its `channel='whatsapp'`, defaults their outbound to WhatsApp; otherwise SMS. Integrates into v4's `enqueue_message` so existing booking-signal-driven outbound automatically routes to WhatsApp for WhatsApp-native customers.

## Acceptance Criteria
- [ ] `backend/outbound/channel_pref.py::pick_channel_for(customer) -> 'sms' | 'whatsapp'` returns the channel of the customer's most-recent `Conversation`, restricted to `('sms', 'whatsapp')`. Defaults to `'sms'` if no inbound conversation exists OR the most recent is `web_chat` / `phone` / `manual_staff`.
- [ ] One indexed query: `Conversation.objects.filter(customer=customer, channel__in=['sms','whatsapp']).order_by('-created_at').values_list('channel', flat=True).first() or 'sms'`.
- [ ] `backend/outbound/api.py::enqueue_message` calls `pick_channel_for(customer)` when its `channel` arg is left at default. Explicit `channel=` kwarg still wins (campaigns slice passes channel explicitly).
- [ ] Tests in `tests/test_channel_preference.py`:
   - Customer with most-recent SMS conversation → defaults to SMS.
   - Customer with most-recent WhatsApp conversation → defaults to WhatsApp.
   - Customer with no inbound history → defaults to SMS.
   - Customer with most-recent web_chat conversation → defaults to SMS (web_chat is not an outbound channel).
   - Explicit `enqueue_message(..., channel='whatsapp')` ignores the lookup.
- [ ] Existing outbound tests pass unchanged (the new branch only fires when `channel` arg is omitted; existing tests pass it explicitly or use the default which was already SMS).

## Technical Details

- The default for `enqueue_message(..., channel='sms')` becomes `channel=None`; if `None`, call the helper. Backwards-compatible because no caller has been passing `channel='sms'` explicitly (they relied on the default).
- `pick_channel_for` is pure Python on the Conversation model — no Django request context needed. Reusable from anywhere.
- Performance: one indexed query per outbound enqueue. The booking signal fires ~2 enqueues per booking → negligible.
