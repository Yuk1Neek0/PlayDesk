# Task: Extract _verify_twilio_signature to shared helper

## Description

Move the Twilio signature-verification logic currently inline in `backend/agent/channels/twilio_sms.py` (and its webhook) into a shared `backend/agent/channels/twilio_signature.py` module. Refactor the existing SMS adapter + webhook to import from there. Pure refactor — no behaviour change. Required by the WhatsApp inbound adapter (task 002), the WhatsApp webhook (task 004), and the voice-scaffold webhook in the parallel `voice-scaffold` slice.

## Acceptance Criteria
- [ ] `backend/agent/channels/twilio_signature.py` exists with `verify_twilio_signature(url: str, post_data: dict, signature: str, auth_token: str) -> bool` — pure function, no Django imports.
- [ ] `backend/agent/channels/twilio_sms.py` imports + calls the new helper; the inline duplicate is removed.
- [ ] `backend/api/webhooks_twilio.py`'s existing `twilio_sms_webhook` continues to verify signatures via the helper (same call shape).
- [ ] Existing 397-test suite passes unchanged. Specifically the existing tests for signed payload / tampered signature / missing token still pass against the SMS webhook.
- [ ] One new unit test in `tests/test_twilio_signature.py` covers the helper directly (signed-OK / tampered-bad / empty-token-bad).

## Technical Details

- The helper signature mirrors what `twilio.request_validator.RequestValidator` expects internally — accept the URL, the POST data dict, the signature header, and the auth token; return bool.
- Do NOT change the SMS adapter's *behaviour* — only the import path. Same 503-on-missing-token, same 403-on-tampered-signature, same 200-on-valid behaviour.
- The voice-scaffold slice's task 001 may extract this same helper if voice ships first. Whichever lands first wins; the other slice just imports.
