# Task: TwilioWhatsAppAdapter (inbound) + tests

## Description

The inbound counterpart to `TwilioSmsAdapter`. Parses Twilio WhatsApp webhook payloads (which arrive with `From: whatsapp:+E.164`) into a `NormalizedMessage`, routes through the same agent loop, returns TwiML reply. Near-mirror of `twilio_sms.py`.

## Acceptance Criteria
- [ ] `backend/agent/channels/twilio_whatsapp.py` defines `TwilioWhatsAppAdapter(ChannelAdapter)` with `channel = "whatsapp"`.
- [ ] `normalize_inbound(payload, headers)` strips the `whatsapp:` prefix from `From`, yielding a plain E.164 `customer_identifier`.
- [ ] `format_outbound(reply_text, metadata)` returns TwiML `<Response><Message>{reply}</Message></Response>` (same shape as SMS).
- [ ] Adapter registry (`backend/agent/channels/registry.py`) gains an inbound entry: `'whatsapp' → TwilioWhatsAppAdapter`.
- [ ] Tests in `tests/test_whatsapp_adapter.py`:
   - Inbound `whatsapp:+14165550111` becomes `+14165550111` in the NormalizedMessage.
   - Inbound carrying a Chinese body preserves the body verbatim (no encoding mangling).
   - `format_outbound("hello")` returns valid TwiML containing `hello`.
   - Registry lookup `get_inbound_adapter("whatsapp")` returns an instance of the class.

## Technical Details

- Pure Python on the ABC (no Django imports) — matches the existing inbound `ChannelAdapter` pattern.
- The `whatsapp:` prefix is the only structural difference from SMS at the message level; reusing the SMS Twilio sandbox for testing requires no extra code beyond this prefix handling.
- Do NOT depend on the WhatsApp webhook (task 004) — that's a separate file. This task is the adapter only.

## Dependencies

- Task 001 (signature helper — used by the webhook in 004, not by this adapter directly, but cleanest to land first).
