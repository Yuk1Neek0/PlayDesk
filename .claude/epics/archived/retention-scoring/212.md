# Task: Admin surfaces + re-engagement template + bulk send

## Description

Brings the data to staff. Cohort filter on the customer list + per-cohort counts chip + a new re-engagement SMS template + a "send this template to all visible" action button + the backing endpoint.

## Acceptance Criteria

- [ ] Backend: `AdminCustomerListView` augmentation:
   - Accepts `?cohort=new|active|at_risk|dormant|lost` query param. Filters the queryset.
   - List response includes a `cohort_counts: {new: N, active: M, ...}` block (store-scoped totals — these are cheap, all from indexed columns).
- [ ] Backend: new `POST /api/admin/customers/bulk-send/`:
   - Body: `{cohort, template_key}`.
   - Validates `template_key` is in an allowlist (currently just `re_engagement_60d` — extensible).
   - Queries customers in that cohort + current store, EXCLUDES customers with `sms_opt_out` in tags, ENQUEUES outbound via the v4 adapter for each.
   - Logs each enqueued message + creates a `CustomerNote(author=request.user, body=f"Sent {template_key} via bulk action")` per customer.
   - Returns `{sent: N, skipped: M, skip_reasons: {...}}`.
   - Gated by v10a's `StaffOnlyMiddleware` automatically (under `/api/admin/*`).
- [ ] `outbound/templates.py` extend:
   - Add `"re_engagement_60d": ("Hi {customer_name}, it's been a while! Come back this week and we'll comp your first hour. Reply YES to book — or skip if you're set.", "{customer_name} 您好，最近没见到您！本周回来，第一小时免费 — 回复 YES 即可预订。")`.
   - Add to `URGENT_TEMPLATE_KEYS`? No — re-engagement is not urgent; respect quiet hours.
- [ ] Frontend: `/admin/customers/` page:
   - Cohort filter dropdown chip in the toolbar. Options: All, New, Active, At risk, Dormant, Lost. Each shows the count.
   - When a cohort other than "All" is selected, show a "Send re-engagement to all {N} visible" button. Click → confirm modal → POST bulk-send → toast "Sent to N customers, skipped M (opt-out)".
- [ ] Frontend: small `cohort-chip.tsx` component for displaying a customer's cohort consistently (colored: new=blue, active=green, at_risk=yellow, dormant=orange, lost=red).
- [ ] Customer detail page (if exists separately): show cohort chip + churn_score + retention_updated_at next to the existing profile fields.
- [ ] Tests:
   - Backend: list filter returns only that cohort + counts correct.
   - Backend: bulk-send enqueues N messages, skips opted-out, creates N CustomerNote rows, returns correct counts.
   - Backend: bulk-send rejects unknown template_key with 400.
   - Frontend: cohort chip renders right color per cohort.
- [ ] `npm run lint && npm run typecheck && npm test && npm run build` clean.

## Technical Details

- `cohort_counts` calculation: a single GROUP BY query: `Customer.objects.filter(store=request.store).values("cohort").annotate(count=Count("id"))`. Cheap.
- Bulk-send: don't rate-limit at the API level (the outbound layer already does that). DO emit one CustomerNote per enqueued message for audit.
- The confirm modal should show the cohort name + count + a sample of the template body so staff don't accidentally blast the wrong message.
- For the cohort-chip colors, use Tailwind utility classes inline — no new design tokens needed.
