# Task: Retention data layer

## Description

Three new fields on `Customer`. Defaults are safe ("new" cohort, 0.0 score). The first run of the sweeper (task #211) backfills accurate values; until then, every customer reads as "new" which is informative-but-not-misleading.

## Acceptance Criteria

- [ ] `core.Customer` augmentation:
   - `churn_score: FloatField(default=0.0, db_index=True)`
   - `cohort: CharField(max_length=16, choices=[("new","New"),("active","Active"),("at_risk","At risk"),("dormant","Dormant"),("lost","Lost")], default="new", db_index=True)`
   - `retention_updated_at: DateTimeField(null=True, blank=True)`
- [ ] Migration `core/migrations/000X_customer_retention_fields.py`:
   - Additive. Explicit `name=` on the indexes (avoid v4 campaigns CI flake).
- [ ] `CustomerSummarySerializer` + `CustomerDetailSerializer` (in `backend/api/serializers.py`) expose the three new fields as read-only.
- [ ] Tests in `tests/test_customer_retention_fields.py`:
   - Existing customers have `cohort="new"` + `churn_score=0.0` + `retention_updated_at=None` after migration.
   - Field defaults applied on `Customer.objects.create(...)`.
   - Indexes exist (Django will create them; sanity-check via `connection.introspection.get_constraints`).
- [ ] Existing 739-test backend suite passes.
- [ ] `python manage.py makemigrations --check` clean.

## Technical Details

- `db_index=True` on both `churn_score` and `cohort` so the admin filter (task #212) is fast on store-scoped queries.
- The cohort choices are deliberately ordered "increasing concern" (new → active → at_risk → dormant → lost) — matches the natural sort.
- `retention_updated_at` is null until first sweeper run; this is the signal "we haven't computed this yet".
