# Task: Cohort + churn-score logic + nightly sweeper

## Description

The pure functions that derive the labels + the management command that runs them across all customers. Standalone — can be exercised before any UI exists.

## Acceptance Criteria

- [ ] `backend/core/retention.py` (new):
   - `compute_cohort(customer, now=None) -> str`:
     ```python
     now = now or timezone.now()
     last = customer.last_visit_at
     created = customer.created_at
     visits = customer.total_visits
     if visits == 0 and (now - created).days < 7:
         return "new"
     if last is None or (now - last).days > 90:
         return "lost"
     days_since = (now - last).days
     if days_since > 60:
         return "dormant"
     if days_since > 30:
         return "at_risk"
     return "active"
     ```
   - `compute_churn_score(customer, now=None) -> float`:
     - Baseline: `days_since / 90.0`, clamped [0, 1]. If `last_visit_at` is None, score = 1.0 (lost).
     - Frequency adjustment: if `total_visits >= 5`, multiply by `min(2.0, total_visits / 10)`. Caps the multiplier at 2× so a 100-visit customer who's been gone 30 days doesn't artificially inflate to score > 1.
     - Final clamp: max(0.0, min(1.0, score)).
- [ ] `backend/core/management/commands/recompute_retention.py`:
   - `Command.handle(**options)`:
     - Iterates `Customer.objects.iterator(chunk_size=500)` to avoid loading all rows into memory.
     - For each: `cohort = compute_cohort(c)`, `score = compute_churn_score(c)`, update if either changed (avoid pointless DB writes), set `retention_updated_at = now()`.
     - Tracks running totals per cohort.
     - At end, logs the distribution + delta vs last run (if available — use the most-recent `retention_updated_at` from before this run as the marker).
   - Options:
     - `--store SLUG` — scope to one store.
     - `--dry-run` — log without writes.
   - Idempotent + safe to re-run.
- [ ] Tests in `tests/test_retention.py`:
   - **compute_cohort**: each threshold (new with <7 days, active <30, at_risk 30-60, dormant 60-90, lost >90, lost with no last_visit_at).
   - **compute_churn_score**: math correct at baseline + with frequency multiplier; clamped at 0/1.
   - **recompute_retention command**: runs over 5 fixture customers covering all cohorts, asserts correct cohort assignment for each, asserts `retention_updated_at` populated.
   - **dry-run**: no DB writes, output shows expected distribution.
   - **--store filter**: only that store's customers updated.
   - **Idempotency**: re-running immediately is a no-op (zero writes — the unchanged-skip check works).
   - **Performance**: synthetic test creates 1000 customers, asserts the command finishes in < 5 seconds.
- [ ] Existing 739-test backend suite passes.

## Technical Details

- `compute_cohort` is pure — no DB queries. The customer's fields are read directly.
- Use `bulk_update` if you want to batch the writes, but the per-row save is fine at v11c scale (likely < 10k customers).
- The "unchanged skip" check: `if c.cohort != new_cohort or abs(c.churn_score - new_score) > 0.01:` — avoids writes when nothing changed.
- For the dry-run, use `print()` or `self.stdout.write()` (Django command idiom).
