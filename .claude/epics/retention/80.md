# Task: Backfill data migration + lookup-or-create on booking creation

## Description

Two things: (1) a data migration that walks existing `Booking` rows, normalises `customer_phone`, and links the booking to a found-or-created `Customer`. (2) `create_booking` (both the agent tool and the REST endpoint) now ALWAYS resolves to or creates a `Customer` row before persisting the booking, and refuses to write a booking with a null `customer_id` after this lands.

## Acceptance Criteria

- [ ] Data migration is idempotent: running it twice does not duplicate customers.
- [ ] After migration, every existing booking has a non-null `customer_id`.
- [ ] A new booking created via the agent or the REST API with a known phone reuses the existing `Customer` row; an unknown phone creates one and the booking links to it.
- [ ] Unparseable phones bubble up as a structured error from `create_booking` (no silent fallback).
- [ ] 6+ tests cover: dedup on second booking, create on first booking, update of blank fields (name/locale_pref) on a sparse existing customer, rejection of bad phone, per-store isolation, agent-tool path.

## Technical Details

- Wrap the lookup-or-create call in `transaction.atomic()` to avoid duplicate inserts under concurrency.
- Conflicts with 79 (both touch `agent_tools/tools.py::create_booking`); land 79 first.
