## Description

Verify the dev plan's backend acceptance criteria (§1.1) against the live stack: booking CRUD, database-enforced overlap rejection, and incremental SSE token emission. Fix any integration bug found, in-stream.

## Acceptance Criteria
- [ ] `curl` round-trip: create a booking (`POST /api/bookings/`), query it (`GET`), modify it (`PATCH`), cancel it (`DELETE`) — all succeed.
- [ ] `GET /api/resources/` and `?type=` filter return correct data; `GET /api/resources/<id>/availability/?date=` computes open slots from business hours minus bookings.
- [ ] Two concurrent `POST /api/bookings/` at the same `(resource_id, time)` yield exactly one `200`/`201` and one `409` — confirmed rejection comes from Postgres, not application code.
- [ ] `GET /api/admin/conversations/` and `GET /api/admin/bookings/` return staff-visible data, bookings sorted by `created_at` desc.
- [ ] The SSE messages endpoint emits assistant tokens incrementally (multiple events over time), not as a single payload.
- [ ] Verification record written to `updates/002/`: criterion → pass/fail → exact command + observed output.
