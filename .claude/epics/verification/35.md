# Task: Stream D — enhancements verification

## Description

Verify the four nice-to-have capabilities (§2.1–2.4) against the live stack: the eval harness, Stripe sandbox deposits, conflict-aware slot suggestions, and bilingual retrieval. Fix any integration bug found, in-stream.

## Acceptance Criteria
- [ ] The eval harness replays its curated conversation set against the **live agent** and reports per-case pass/fail plus aggregate accuracy.
- [ ] Stripe deposit flow with **real test-mode keys**: `create_booking` initiates Checkout; on payment success the webhook moves `Booking.status` from `pending_payment` to `confirmed`.
- [ ] A `pending_payment` booking expires after its TTL via the `expire_holds` periodic command.
- [ ] `check_availability` returns nearby `suggestions` (`{ "available": [], "suggestions": [...] }`) when the requested slot is taken.
- [ ] A Chinese-language message retrieves `lang=zh` `KnowledgeChunk`s and the agent replies in Chinese; an English message does the English equivalent.
- [ ] Verification record written to `updates/005/`: criterion → pass/fail → command + evidence.

## Technical Details

- Stripe: load test-mode keys from env; run `stripe listen` to forward webhooks to the local backend; complete a test-mode Checkout and observe the status transition.
- TTL expiry: create a `pending_payment` booking, run `expire_holds`, confirm it expires.
- Slot suggestions: book a slot, then request the same slot and assert the `suggestions` shape.
- Bilingual: send one EN and one ZH message; verify retrieved chunk `lang` and reply language.
- Primary fix surface: `backend/evals/`, `backend/core/payments.py`, `expire_holds`, slot-suggestion and language code paths.

- Stripe test-mode keys + `stripe` CLI available.
