# Task: Stream B — agent loop & RAG verification

## Description

Verify the dev plan's AI-core and RAG acceptance criteria (§1.3, §1.4, §1.5) against the live agent making real LLM calls: natural-language booking, RAG-vs-SQL routing, the persisted reasoning trace, and the iteration-cap fallback. Fix any integration bug found, in-stream.

## Acceptance Criteria
- [ ] A single message — *"Saturday 8pm, PS5 for 3 people, around 2 hours"* — drives the agent to complete a booking.
- [ ] RAG-vs-SQL routing: *"Can I bring outside food?"* is answered via `search_knowledge_base`; *"Is room 3 free at 8pm tomorrow?"* via `check_availability` — never the reverse.
- [ ] Smoke conversations show the LLM selecting the correct tool per query type (`get_resource_details`, `create_booking`, `modify_booking`, `cancel_booking`).
- [ ] The `Message` table after a conversation holds the full turn-by-turn trace (user / assistant / tool call / tool result) with tool payloads + results in JSONB, and reads coherently end-to-end.
- [ ] The 6-iteration cap triggers the graceful human-handoff fallback message.
- [ ] Tool execution failures return a structured error to the LLM rather than raising to the user.
- [ ] Verification record written to `updates/003/`: criterion → pass/fail → conversation + trace evidence.

## Technical Details

- Drive conversations through the live messages endpoint; inspect resulting `Message` rows directly in Postgres.
- Routing check confirms the system prompt's structured/unstructured partition holds in practice.
- Iteration cap: construct or simulate a conversation that loops to the 6-iteration limit.
- Primary fix surface: `backend/agent/`, `backend/rag/`, `backend/agent_tools/`. Flag any shared-file fix for Task 006.

## Dependencies

- Task 001 (stack booted, KB ingested, LLM key present).
