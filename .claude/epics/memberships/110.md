# Task: Admin endpoints — membership view, adjust-points, redeem, rewards CRUD, tiers CRUD

## Description

The DRF surface that powers the entire admin frontend. Five endpoints: a composite "membership view" for the customer-detail page, a staff-adjustment endpoint with required reason, an atomic redemption endpoint, plus standard CRUD ViewSets for rewards and tiers.

All endpoints are admin-only — reuse the existing admin auth gate the verification epic put in place.

## Acceptance Criteria
- [ ] `GET /api/admin/customers/{id}/membership/` returns `{ balance, lifetime_earned, tier: {name, perks_text} | null, next_tier: {name, min_lifetime_points} | null, points_to_next_tier, recent_transactions: [...up to 20], available_rewards: [...] }` in <150ms for a customer with 10k transactions.
- [ ] `POST /api/admin/customers/{id}/adjust-points/` accepts `{ delta: int, reason: str }`; rejects empty reason with 400; writes one transaction with `source="adjustment"`, `reference=reason`, `author=request.user`.
- [ ] `POST /api/admin/customers/{id}/redeem/` accepts `{ reward_id: int }`; atomic — locks customer, verifies `balance >= reward.cost_points`, writes the debit transaction and the `Redemption` row in one DB transaction; returns updated `{ balance, redemption_id }`. Returns 409 `{"error":"insufficient_points","balance":N,"cost":M}` if check fails.
- [ ] `/api/admin/rewards/` ModelViewSet: list / create / retrieve / partial_update / destroy. List is store-scoped to the request user's accessible stores.
- [ ] `/api/admin/tiers/` ModelViewSet: same shape, store-scoped.
- [ ] All endpoints behind the admin auth gate (verified by a test that confirms an unauthenticated request returns 403).
- [ ] Tests: happy-path read; happy-path redeem; insufficient-points redeem returns 409 and writes no rows; adjustment with empty reason returns 400; adjustment with valid reason writes one transaction with the correct author; rewards / tiers CRUD round-trip; reward/tier list does not leak cross-store rows.

## Technical Details

- File: `backend/api/views_memberships.py` (new).
- URL conf: extend `backend/api/urls.py` with the new routes.
- The composite endpoint is built with `Prefetch` / `select_related` to avoid N+1 — one query for the latest transaction + one for the 20 most-recent + one for available rewards + one for tier lookup.
- The redeem endpoint wraps its body in `transaction.atomic()` and calls `award_points(customer, -reward.cost_points, "redemption", str(redemption.id))` followed by `Redemption.objects.create(...)`. Because `award_points` uses `select_for_update`, two concurrent redeem calls on the same customer cannot double-spend.
