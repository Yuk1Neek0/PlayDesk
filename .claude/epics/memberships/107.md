# Task: award_points() + tier_for() + memberships_check command + invariant tests

## Description

Build the single ledger-write helper that all earn / redeem / adjustment paths funnel through, plus the pure-function tier resolver, plus the `memberships_check` management command that asserts `SUM(delta) == latest.balance_after` for every customer. Cover with the unit tests that prove the helper is concurrency-safe and the tier resolver behaves correctly at every edge.

This is the hot path — every other task that writes points routes through it. Get it right once, then never re-implement it.

## Acceptance Criteria
- [ ] `core.memberships.award_points(customer, delta, source, reference, author=None) -> PointTransaction` writes one row and returns it.
- [ ] The function acquires `select_for_update()` on the customer row before reading `balance_after` from the latest transaction — two concurrent calls on the same customer serialise correctly (covered by a multi-thread test).
- [ ] `core.memberships.lifetime_points_earned(customer) -> int` returns `SUM(delta) WHERE delta > 0`.
- [ ] `core.memberships.tier_for(customer) -> RewardTier | None` returns the tier with the highest `min_lifetime_points <= lifetime_points_earned`, or `None` if the customer is below the lowest tier or no tiers are configured for the store.
- [ ] `python manage.py memberships_check` iterates every customer, asserts `SUM(delta) == latest.balance_after`, exits 0 on success / 1 on any discrepancy (with the offending customer ids logged).
- [ ] CI runs `memberships_check` on a seeded test dataset.
- [ ] Tier resolver covered by ≥5 unit tests: empty balance, exact-threshold, between-tiers, above-top-tier, no-tiers-configured.
- [ ] `award_points` covered by tests for: positive delta, negative delta, zero delta rejection, concurrent calls preserve `SUM == balance_after`.

## Technical Details

- File: `backend/core/memberships.py` (new).
- The concurrency test uses Django's `transaction.atomic` + threading to fire two `award_points(customer, +10, ...)` calls and asserts the customer ends at the correct balance with two rows whose `balance_after` are sequential.
- `tier_for` is called from the customer-detail render and from the public QR page header — it MUST NOT issue an unindexed query. Implementation: one `Tier.objects.filter(store=customer.store).order_by('-min_lifetime_points')` + Python-side comparison; bounded by tier count, not customer count.
- The `memberships_check` command's failure exit code is what catches drift; the test asserts both happy + drifted-dataset paths.
