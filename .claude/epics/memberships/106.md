# Task: Migration — PointTransaction + Reward + Redemption + RewardTier + Store.points_per_booking

## Description

Add the schema for the entire memberships module in one additive migration. Includes the append-only ledger (`PointTransaction`), the per-store reward catalogue (`Reward`), the audit row (`Redemption`), the tier definitions (`RewardTier`), and the new store-level config field (`Store.points_per_booking`).

This is the foundation task — every other task in the epic depends on this. Run first, in isolation.

## Acceptance Criteria
- [ ] Migration applies clean forward (`migrate`) and reverse (`migrate core <previous>`).
- [ ] `PointTransaction` has fields: `customer (FK CASCADE)`, `delta (int, signed)`, `source (CharField, choices: booking|qr_click|redemption|adjustment|backfill)`, `reference (CharField max 100)`, `author (FK AUTH_USER_MODEL, nullable SET_NULL)`, `created_at (auto_now_add)`, `balance_after (int)`.
- [ ] `Reward` has: `store (FK CASCADE)`, `name`, `description (text)`, `cost_points (PositiveIntegerField, MinValueValidator(1))`, `enabled (bool default True)`, `created_at`.
- [ ] `Redemption` has: `customer (FK CASCADE)`, `reward (FK PROTECT)`, `transaction (OneToOneField PointTransaction CASCADE)`, `staff (FK AUTH_USER_MODEL, SET_NULL)`, `redeemed_at (auto_now_add)`.
- [ ] `RewardTier` has: `store (FK CASCADE)`, `name`, `min_lifetime_points (PositiveIntegerField)`, `perks_text (text)`, `position (PositiveIntegerField)`. Unique on `(store, position)`.
- [ ] `Store.points_per_booking` added with `default=10`.
- [ ] Indexes: `(customer, -created_at)` on `PointTransaction` (powers the "last 20 transactions" query); `(store, enabled)` on `Reward`.
- [ ] `verbose_name` set on `id` fields (matches the project's foundation-epic convention).

## Technical Details

- File: `backend/core/migrations/000Z_memberships.py` — exact migration number determined by `python manage.py makemigrations`.
- Model classes added to `backend/core/models.py` near the existing `Customer` / `QRAction` blocks.
- `PointTransaction.balance_after` is denormalised for fast read but is NOT updated by the migration — populated by `award_points()` (task 002) and the backfill (task 004).
- No data migration in this task — purely additive schema.
