# Task: Earn signals — booking-completion + identified QR-click hook

## Description

Wire the two earn paths into already-shipped surfaces. (1) Add a `post_save Booking` signal that calls `award_points` on transitions to `status = "completed"`, crediting `customer.store.points_per_booking`. (2) Extend the existing QR-event endpoint so that an identified click on an action with `reward_points > 0` also calls `award_points`.

Both paths share one rule: anonymous events earn nothing. The signal and the QR hook both check for `customer is not None` first.

## Acceptance Criteria
- [ ] `post_save Booking` signal in `backend/core/signals.py` calls `award_points(booking.customer, booking.store.points_per_booking, "booking", str(booking.id))` when the booking transitions to `status = "completed"`.
- [ ] The signal is idempotent — re-saving an already-completed booking (e.g. a status touch) does not award points twice. Implementation: check for an existing `PointTransaction` with `source="booking"` and `reference=str(booking.id)`.
- [ ] The signal is a no-op if `booking.customer is None` (legacy bookings pre-retention backfill).
- [ ] The existing QR-event endpoint (`backend/api/views_qr.py`, the view handling `POST /api/qr/event/`) — after recording a click `QREvent` — calls `award_points(event.customer, event.action.reward_points, "qr_click", str(event.id))` iff `event.customer is not None AND event.action is not None AND event.action.reward_points > 0`.
- [ ] Tests cover: booking transitions to completed → +points; booking re-saved while already completed → no duplicate; booking without customer → no-op; QR click while identified → +points; QR click anonymous → no points; QR click on action with `reward_points=0` → no transaction row.

## Technical Details

- Files touched: `backend/core/signals.py` (extend), `backend/api/views_qr.py` (extend the existing event POST handler), `backend/core/apps.py` (ensure the signal is connected on app ready if not already).
- The QR hook is a fire-and-forget addition after the existing event-write — it MUST NOT block or change the response shape of `POST /api/qr/event/` (the existing one-qr success criterion says the endpoint never blocks the redirect).
- Wrap the QR hook's `award_points` call in a `try/except Exception: logger.exception(...)` so an awarding error never breaks click tracking.
