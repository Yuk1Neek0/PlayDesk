## Acceptance Criteria
- [ ] `backend/outbound/signals.py` connects a `post_save` signal on `Booking`:
  - **On create**: enqueues `booking_confirmation` (`scheduled_for = now()`), reference `"booking:<id>:confirm"`.
  - **On create**: enqueues `reminder_24h` (`scheduled_for = booking.scheduled_at - timedelta(hours=24)`), reference `"booking:<id>:reminder_24h"`. Only enqueued if the scheduled time is in the future at creation time.
  - **On transition to `status='no_show'`**: enqueues `no_show_followup` immediately, reference `"booking:<id>:no_show"`.
  - **On transition to `status='completed'`**: enqueues `booking_thank_you` immediately, reference `"booking:<id>:thank_you"`.
  - **On transition to `status='cancelled'`**: bulk-updates all `OutboundMessage` rows with `reference LIKE 'booking:<id>:%'` and `status='queued'` to `status='cancelled'`.
- [ ] All signal handlers are no-ops if `booking.customer is None` (legacy bookings).
- [ ] The signal relies on `enqueue_message`'s idempotence â€” re-saving an already-completed booking does not double-enqueue.

## Technical Details

- File: `backend/outbound/signals.py`. Wired in `backend/outbound/apps.py::OutboundConfig.ready()`.
- The transition detection uses `pre_save` to capture the prior status, compared against `post_save`'s new status. Alternative: a `dispatch_uid` per template_key + a manual status check in `post_save`. Either is acceptable.
