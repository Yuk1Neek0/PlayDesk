# Task: send_outbound management command + quiet hours + opt-out enforcement + cron doc

## Description

The cron-driven sender that drains the `OutboundMessage` queue. Selects due `queued` rows, looks up the adapter via the registry, sends, updates row status. Respects quiet hours (reschedules) and opt-out (cancels). Documented for operator wiring.

## Acceptance Criteria
- [ ] `python manage.py send_outbound` selects `OutboundMessage` rows where `status='queued' AND scheduled_for <= now()`, ordered by `scheduled_for ASC`, capped at 200 per run.
- [ ] Uses `SELECT ... FOR UPDATE SKIP LOCKED` (`select_for_update(skip_locked=True)`) so two concurrent invocations never grab the same row.
- [ ] For each row:
  - If `"sms_opt_out" in customer.tags`: mark `cancelled`, `failure_reason = "opt_out"`, continue.
  - If quiet hours apply (and the template is not `booking_confirmation`): update `scheduled_for` to the next allowed boundary, leave `status = queued`, continue.
  - Otherwise: call the registered adapter's `send()`, update the row in one transaction. On `ok=True`: `status='sent'`, `sent_at=now()`, optionally store `provider_message_id` in a new `provider_message_id` field (add via this task's migration or via task 001's migration â€” decide in implementation).
  - On `ok=False` with `reason='not_configured'`: leave `status=queued`, log `[outbound] skipped: twilio not configured` once per run.
  - On `ok=False` with any other reason: `status='failed'`, `failure_reason=reason`.
- [ ] `backend/outbound/quiet_hours.py::next_send_time(scheduled, store, urgent=False) -> datetime` returns the original time if outside quiet hours OR `urgent=True`; otherwise returns the next `store.quiet_hours_end` instant in store-local time.
- [ ] `urgent` is True for `template_key == "booking_confirmation"` and False otherwise.
- [ ] `docs/outbound-cron.md` documents: cron command, recommended cadence (every 60s), systemd-timer example, Docker-sidecar example, what to monitor.
- [ ] Tests:
  - Empty queue â†’ exits 0 with no log spam.
  - Mixed queue (due / future / opted-out) â†’ only due-and-not-opted-out rows get sent.
  - Concurrent invocation (two threads, same queue) â†’ each row processed at most once.
  - Quiet-hours rescheduling â†’ row stays `queued`, `scheduled_for` is updated to the next allowed moment.
  - `booking_confirmation` inside quiet hours â†’ still sent (urgent bypass).
  - `not_configured` adapter result â†’ row stays `queued`, no transition to `failed`.
  - Generic adapter failure â†’ row transitions to `failed` with `failure_reason` set.

## Technical Details

- File: `backend/outbound/management/commands/send_outbound.py`.
- Concurrency: per-row processing in its own `transaction.atomic()` block; the outer iteration is not transactional (long-running transactions are bad practice for cron-jobs).
- Quiet-hours math is store-local: convert `now()` to `store.timezone`, compare against `store.quiet_hours_start` / `_end`. Handles wrap-around (e.g. 22:00 â†’ 08:00 crossing midnight).
- Logging: structured-friendly key=value style matching existing logs.
- Cron doc must include an example for both bare-metal and Docker â€” repo runs via `docker compose`, but staging might run elsewhere.
