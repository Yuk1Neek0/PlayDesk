# Task: Agent tool contracts & registry

## Description

Define the six agent-tool interfaces as Pydantic schemas plus a tool registry, all stubbed to return representative data. This freezes the **tool-interface contract** so the agent loop and each real tool can be built independently in Wave 1.

## Acceptance Criteria
- [ ] Pydantic input/output schemas for all six tools: `search_knowledge_base`, `check_availability`, `get_resource_details`, `create_booking`, `modify_booking`, `cancel_booking`.
- [ ] `check_availability` output schema carries both `available` and `suggestions` arrays (slot-suggestion hook).
- [ ] `create_booking` output models either a `booking_id` success or a structured conflict error.
- [ ] A tool registry exposes the tools by name with their schemas (the agent-loop ↔ tool boundary).
- [ ] Each tool is a stub returning representative fake data; signatures are final.
- [ ] A test confirms every tool is discoverable via the registry and its stub validates against its schema.

## Technical Details
- Live under `backend/` (e.g. a `tools/` package) — schemas, registry, stubs only; no DB access yet.
- Schemas are the contract; Wave 1 swaps stub bodies for real implementations without touching signatures.
