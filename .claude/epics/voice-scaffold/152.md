# Task: docs/voice-implementation-plan.md with full design coverage

## Description

The deeper design doc that picks up where `docs/voice-ai-readiness.md` (v3 multi-channel) left off. Contains the vendor recommendation, latency budget, cost ceiling, partial-transcript wiring, fallback behaviour, and "what to build first." Sized so a future engineer (human or agent) can start implementation without further design work.

## Acceptance Criteria
- [ ] `docs/voice-implementation-plan.md` exists with these sections (all populated, no `TODO` placeholders):
   - **Overview** — one paragraph stating the goal: STT → agent loop → TTS round-trip over Twilio Voice.
   - **Vendor stack recommendation** — one named option (e.g. Twilio Media Streams + Deepgram streaming STT + ElevenLabs streaming TTS), with rationale, and one alternative.
   - **Mermaid sequence diagram** — caller → Twilio → STT → agent → TTS → caller, with the partial-transcript and barge-in points marked.
   - **Per-step latency budget** — target round-trip latency, broken down: STT first-token, agent first-token, TTS first-audio, network round-trip. Should add up to a target ≤ 1500ms first-audio.
   - **Per-call cost ceiling** — rough USD/minute estimate for the recommended stack; the implication for which questions should auto-route to voice vs. push the caller back to text.
   - **Partial-transcript wiring** — where Deepgram's WebSocket plugs into the agent loop (`AgentLoop.run` currently takes a complete user message; what changes to make it consume interim transcripts).
   - **Fallback behaviour** — STT/TTS provider failures, human-handoff path, dropped-call recovery (using v4 outbound to text the caller back).
   - **"What to build first"** — three concrete next steps, in order, with file paths.
- [ ] Cross-references the existing `docs/voice-ai-readiness.md` (don't duplicate; this is the next layer down).
- [ ] Cross-references the v5 voice-scaffold webhooks (task 002, task 003) as the current entry point.
