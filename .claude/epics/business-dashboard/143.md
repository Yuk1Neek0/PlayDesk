# Task: today_local(store) helper + tests

## Description

Small date helper that returns "today" in a store's local timezone — the foundation under `bookings_today` in the business-metrics endpoint. The v3 timezone fixes established that all display-side time math runs in store-local time; this slice needs that for the day-window aggregate too.

## Acceptance Criteria
- [ ] `backend/core/dates.py::today_local(store: Store) -> date` returns the current date in `store.timezone` (e.g. `America/Toronto`).
- [ ] If `store.timezone` is empty or invalid, falls back to UTC and logs a warning (matches v3 retention's defensive posture).
- [ ] Tests in `tests/test_dates.py`:
   - At a UTC time that's the day BEFORE the store-local day (e.g. UTC 03:00 with store at America/Toronto = previous day) → returns the store-local date, not UTC's date.
   - At a UTC time that's the day AFTER the store-local day (early-evening UTC = same day Toronto) → returns the store-local date.
   - With `store.timezone = "Invalid/Timezone"` → returns UTC date, logs warning.
- [ ] Tests use `freezegun` or equivalent to pin the wall clock so they're deterministic.

## Technical Details

- The helper uses `zoneinfo.ZoneInfo` (already used elsewhere in the project) to convert UTC `now()` to the store's local time, then strips to `.date()`.
- A fall-through to UTC on invalid timezone is safer than raising — the dashboard endpoint should not 500 because of a single misconfigured store.
- If `core/dates.py` already exists, extend it; otherwise create.
