# Task: Backend view sweep — every admin view filters by request.store

## Description

The mechanical-but-essential sweep that makes multi-location actually work on the data layer. Every place that currently does `Store.objects.first()` becomes `request.store`. Every admin queryset gains a `.filter(store=request.store)` (or equivalent FK chain). Plus a new `GET /api/admin/stores/` endpoint that the frontend store-switcher (#004) needs to populate its chip group. The cross-store isolation invariant test lands here so the rest of the epic can rely on it.

## Acceptance Criteria
- [ ] `GET /api/admin/stores/` returns `[{id, slug, name}]` for all stores. Live behind whatever the project's standard admin auth is (currently no API-level perm — matches v5 memberships PR #129's convention).
- [ ] Replace `Store.objects.first()` with `request.store` everywhere it currently appears in the API layer. Audit at minimum:
   - `backend/api/views.py` — `AdminBookingListView`, `AdminCustomerListView`, `AdminCustomerDetailView`, `AdminCustomerNoteCreateView`, `AdminConversationListView`, `BookingListCreateView`, `ResourceListView`, etc.
   - `backend/api/views_memberships.py` — membership view, adjust, redeem, rewards/tiers CRUD, `QRTierBadgeView`.
   - `backend/api/views_outbound.py` — outbound message list.
   - `backend/api/views_campaigns.py` — segments + campaigns + runs + send + cancel.
   - `backend/api/views_metrics.py` — `BusinessMetricsView` (every aggregate gains a `store=request.store` filter).
   - `backend/api/views_public.py` — `StoreBrandView` (uses `request.store` which the resolver picks from URL kwarg in v6's `/s/[slug]/...` route, or falls back).
   - QR action / event views.
- [ ] All admin querysets gain `filter(store=request.store)` (or equivalent — `filter(customer__store=request.store)` for Customer-derived models, etc.).
- [ ] `AdminCustomerDetailView` returns 404 instead of 200 when the requested customer is from a different store than `request.store`. Same for `AdminBookingDetailView`, `CampaignDetailView`, etc. (No information-leak via 200 + empty payload.)
- [ ] Cross-store isolation test: `tests/test_multi_store_isolation.py` creates customers + bookings + outbound rows in two stores; asserts each admin view scoped to store A returns ONLY store A rows; asserts 404 on cross-store detail access.
- [ ] Agent tools (`check_availability`, `get_resource_details`, `create_booking`, `modify_booking`, `cancel_booking`) — when called with the explicit `store_id` arg from task 002, all queries scope to that store.
- [ ] Existing 485-test backend suite passes (single-store fixtures continue to resolve to the default store).
- [ ] ≥6 new tests cover the cross-store isolation invariant across the major admin areas (bookings, customers, conversations, outbound, campaigns, memberships).

## Technical Details

- This is a large mechanical task across ~10 view files. Do it methodically: file by file, run the test suite after each, commit per file or per logical group.
- The pattern: `qs = Booking.objects.all()` → `qs = Booking.objects.filter(store=self.request.store)`. For Customer-derived models: `filter(customer__store=self.request.store)`.
- The 404-on-cross-store rule: easiest via `get_object_or_404(MyModel, pk=pk, store=self.request.store)` instead of the bare `get_object_or_404(MyModel, pk=pk)`.
- `views_public.py` is the trickiest because it's public (no auth, no admin context). In v6, it reads `request.store` from the URL kwarg (set by the customer-facing route in task 005) or falls back. The existing `Store.objects.first()` call gets replaced by `request.store` directly.
- The cross-store test is the new invariant tests. Spend the most assertion budget here.
