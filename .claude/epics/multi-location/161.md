# Task: Frontend store-switching surface

## Description

Adds the React context + cookie/localStorage persistence + the visible store-switcher chip in the admin nav. Every existing admin page's `fetch` calls go through a thin `adminFetch` wrapper that auto-injects the `X-PD-Store-Slug` header so the backend resolver picks the right store. Replaces the hardcoded "DOWNTOWN Â· TORONTO" string with the current store's name.

## Technical Details

- The `adminFetch` wrapper is intentionally separate from the existing `request()` helper in `api.ts` â€” `request()` already exists and is used by typed clients; `adminFetch` is the lower-level shape. Either rename `request()` to call `adminFetch` internally, or have typed clients call `adminFetch` directly. Implementation choice; document it in the code.
- Cookie write from JS: `document.cookie = 'pd_store_slug=...; path=/; SameSite=Lax'`. No `Secure` in dev.
- The store-context-aware refetch happens because every admin page's data-fetch effect should already have its `customerId` or page-id in its `useEffect` dependency array; we add `useCurrentStore().current.slug` to those dependency arrays so a store change triggers a refetch.
- Hide the switcher when `stores.length === 1` â€” single-store deployments don't need it.
