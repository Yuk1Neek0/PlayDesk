# Task: Conversation.store FK + backfill + agent loop store-awareness

## Description

Add the `Conversation.store` FK that the agent loop will use to scope its tools. Backfill from existing data using the heuristic in the PRD. Update inbound webhooks to set `store` at conversation creation. Update the agent loop to pass `conversation.store` to tool calls.

## Acceptance Criteria
- [ ] `Conversation.store` FK added (nullable in the schema migration; populated by the data migration; PROTECT on delete).
- [ ] Migration `000X_conversation_store.py` — additive schema.
- [ ] Data migration `000Y_backfill_conversation_store.py`:
   - For each `Conversation` with `customer_identifier` matching a `Customer.phone` (via `normalize_phone`): set `store = customer.store`.
   - For each remaining `Conversation`: set `store = Store.objects.order_by('slug').first()` (the legacy single store).
   - After backfill, an `AlterField` migration (`000Z_conversation_store_required.py` OR same data-migration file with a `null=False` AlterField at the end) flips the field to `null=False`.
- [ ] Inbound Twilio SMS webhook (`twilio_sms_webhook`) — when creating a Conversation, sets `store = request.store` (which the v6 resolver picks based on the matched Customer's store via the lookup-or-default logic).
- [ ] Inbound Twilio WhatsApp webhook (`twilio_whatsapp_webhook`) — same.
- [ ] Inbound Twilio Voice webhook (`twilio_voice_webhook` + `twilio_voice_status_callback`) — same.
- [ ] Web-chat conversation-create view (`POST /api/conversations/`) — sets `store = request.store` (which the resolver picks from the request's header/cookie/URL).
- [ ] `AgentLoop.run` passes `conversation.store` to tool dispatch (either as a method param or by setting it as `loop.current_store`).
- [ ] All 6 agent tools that touch store-scoped models (`check_availability`, `get_resource_details`, `create_booking`, `modify_booking`, `cancel_booking`, `get_booking`) accept an explicit `store_id` argument; current single-store callers default to `Store.objects.first()` for backward compat.
- [ ] Tests:
   - Migration applies clean forward + reverse.
   - Backfill: customer-matched conversations get the right store; unmatched get the default.
   - SMS / WhatsApp / Voice / web-chat webhooks set `Conversation.store` correctly.
   - Agent loop passes the right `store_id` to a mocked `check_availability` tool call.
   - Existing 485-test backend suite still passes (single-store fixtures resolve to the default).

## Technical Details

- The schema migration adds `store = models.ForeignKey(Store, on_delete=models.PROTECT, null=True, related_name='conversations')`. `null=True` initially so the data migration can run.
- The data migration runs in one pass. Use `apps.get_model` for `Customer`, `Conversation`, `Store` (canonical historical-model pattern).
- After the data migration runs, a final `AlterField(null=False)` step pins the constraint. Can be the same file or a separate `000Z_*` migration — either is acceptable.
- Agent-loop test: monkeypatch `check_availability` to capture its `store_id` kwarg; assert it matches `conversation.store_id`.
- Tools' `store_id` arg should have a sensible default (`None` → fall back to first store) so existing callers don't break.
