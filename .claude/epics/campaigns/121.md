# Task: Segment evaluator + DSL parser + store-scoping + unit tests

## Description

The four-key DSL evaluator that compiles a `Segment.filter` JSON into Django ORM `Q` predicates. Always store-scoped by construction. Unknown keys are logged and ignored (forward-compat). No HTTP layer needed for this task — it's testable as a pure function.

## Acceptance Criteria
- [ ] `backend/campaigns/segments.py::customers_for(segment: Segment) -> QuerySet[Customer]`:
  - Always filters by `segment.store` first (cross-store leak is impossible by construction, not by per-call check).
  - Composes `Q` predicates server-side; never loads-then-filters.
- [ ] Supported DSL keys (all optional, combined with AND):
  - `tags_include: list[str]` → `tags__contains=tag` for each (all must be present).
  - `min_total_visits: int` → `total_visits__gte=N`.
  - `last_visit_within_days: int` → `last_visit_at__gte=now() - timedelta(days=N)`.
  - `locale_pref: 'en' | 'zh'` → `locale_pref=...`.
- [ ] Empty filter dict → all customers in the store.
- [ ] Unknown keys → logged at WARNING level (`[campaigns] unknown segment key: <key>`) and silently ignored.
- [ ] Tests:
  - Empty filter returns the full store customer list (and excludes other stores' customers).
  - Cross-store invariant: a segment on store A with the broadest filter never returns a customer from store B (separate test, deliberately set up with two stores).
  - Each DSL key individually filters correctly.
  - Combined DSL keys AND together correctly.
  - `tags_include` with 2+ tags requires all tags (not any).
  - Unknown key logs a warning and does not raise.
  - Boundary: `min_total_visits = 0` matches everything; `last_visit_within_days = 0` matches only today's visitors.

## Technical Details

- File: `backend/campaigns/segments.py`.
- The store-scoping invariant is enforced by always starting the chain with `Customer.objects.filter(store=segment.store)` — never with `Customer.objects.all()`. The test that proves this is the cross-store test.
- `tags_include` requires JSON-list-contains semantics; Django supports this on Postgres with `JSONField`'s `contains` lookup: `Customer.objects.filter(tags__contains=[tag])`. Chained per tag.
- Performance note: `last_visit_at` is already indexed (from retention). `total_visits` is not — if perf testing under load shows it matters, a follow-up task can add a partial index.
