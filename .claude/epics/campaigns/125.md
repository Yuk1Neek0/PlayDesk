# Task: Admin frontend — /admin/segments + /admin/campaigns + confirmation modal

## Description

Five new pages / components: segments list, segment builder, campaigns list, campaign new-flow (segment picker → preview → body → schedule → confirmation), campaign detail (runs table with status filter). The confirmation modal before send is the only thing standing between a curious click and 1000 SMSes — wire it carefully.

## Acceptance Criteria
- [ ] `frontend/src/app/admin/segments/page.tsx` lists existing segments (name, match count via preview, created_by, created_at) with a "New segment" button.
- [ ] `frontend/src/components/admin/segment-builder.tsx` — modal / drawer with: name input; tags chip selector; `min_total_visits` number input; `last_visit_within_days` dropdown (7 / 30 / 60 / 90 / 365 / Any); `locale_pref` dropdown (Any / EN / 中文). Live preview count via debounced (300ms) calls to `/preview/`.
- [ ] `frontend/src/app/admin/campaigns/page.tsx` lists campaigns (name, status chip, segment name, scheduled_for, recipient_snapshot_count, sent_at) with a "New campaign" button.
- [ ] `frontend/src/app/admin/campaigns/new/page.tsx` is a four-step flow: (1) pick segment with live count; (2) draft body in a textarea with `{customer.name}` helper hint; (3) pick schedule (now / pick datetime); (4) confirmation modal showing exact recipient count + body excerpt rendered against the first sample customer.
- [ ] The confirmation modal's "Send" button is only enabled when `count > 0 AND body.trim().length > 0 AND confirm-checkbox is checked`. Submits `POST /api/admin/campaigns/{id}/send/` with `{"confirm": true}`. Shows backend error inline on 4xx.
- [ ] `frontend/src/app/admin/campaigns/[id]/page.tsx` — header with status chip, totals, sent_at, sent_by; paginated runs table with status filter chips (All / Sent / Failed / Skipped opt-out); refresh button.
- [ ] All new pages honour `prefers-reduced-motion`.
- [ ] `npm run lint && npm run typecheck && npm run build` clean.

## Technical Details

- Files added: `frontend/src/app/admin/segments/page.tsx`, `frontend/src/app/admin/campaigns/page.tsx`, `frontend/src/app/admin/campaigns/new/page.tsx`, `frontend/src/app/admin/campaigns/[id]/page.tsx`, `frontend/src/components/admin/segment-builder.tsx`, `frontend/src/components/admin/campaign-confirm-modal.tsx`.
- Reuse existing `pd-button`, `pd-card`, `pd-modal`, `pd-pill` tokens.
- The new-campaign flow stores draft state in React state (no persistence across navigation) — the campaign is only created on Send. This avoids orphan drafts in v4.
- Confirmation modal renders the body against the first sample customer client-side using a tiny `safeFormat(template, ctx)` helper (mirrors the backend's `SafeFormatter` semantics — unknown keys produce a visible `{unknown}` token in the preview so the user spots them before sending).
