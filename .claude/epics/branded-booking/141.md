# Task: QR page refactor onto the helper (behaviour-preserving)

## Description

Refactor `app/qr/[slug]/page.tsx` so its brand read goes through `fetchStoreBrand()` instead of the inline SSR fetch it shipped with in v3. Pure refactor — no rendering change, no new tests. The Playwright `v4-cross-slice.e2e.ts` and `admin.e2e.ts` suites must pass unchanged.

## Acceptance Criteria
- [ ] `frontend/src/app/qr/[slug]/page.tsx` no longer issues its own brand fetch. Uses `fetchStoreBrand()` from `lib/store-brand`.
- [ ] The QR page's rendering (logo, accent, header text) is byte-identical to before for the same `Store.brand` data.
- [ ] All existing Playwright e2e tests (10 currently) pass unchanged.
- [ ] `npm run lint && npm run typecheck && npm run build` clean.
- [ ] No new tests — the refactor's correctness is proved by the existing e2e suite continuing to pass.

## Technical Details

- The QR page's slug parameter does NOT affect the brand fetch in v5 (single-store). When v6 multi-location lands, the helper signature evolves to `fetchStoreBrand(slug?: string)` — this refactor keeps the QR page ready for that change without forcing it now.
- If the existing inline fetch has any QR-specific logic (e.g. a fallback that's narrower than what the helper does), preserve that logic at the call site, NOT in the helper.
