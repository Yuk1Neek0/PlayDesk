# PRD: case-study

## Executive Summary

After streams A–D land, the codebase is rich but unnarrated. This stream is the tail of v2 and turns the shipped system into an interview artifact: a polished `README.md`, an `architecture.md` with a Mermaid diagram, an `ai-design-decisions.md` capturing the load-bearing choices, an `ai-quality-report.md` snapshot generated from Stream C, the `voice-ai-readiness.md` from Stream D, a `demo-script.md` for the 3-minute and 10-minute interview cuts, and a `screenshots/` directory captured against the polished UI.

This is the only sequential stream — it depends on A/B/C/D having merged so the screenshots and the architecture diagram reflect reality, not aspirations.

## Problem Statement

A reviewer dropped into the repo today sees scattered PRDs, archived epics, and code without a single load-bearing entry point. The "demo flow" is in the author's head. The differentiating decisions — database as source of truth, RAG-vs-SQL partition, human handoff, prompt versioning, channel adapters — are visible in the code but not foregrounded.

The COSReady interview window is short. The reviewer needs to understand PlayDesk in 5 minutes and see the demo in 3.

## User Stories

- **As a reviewer (interviewer)**, I open the repo's `README.md` and understand the project's purpose, architecture, and demo flow in 5 minutes.
  - *Acceptance:* a hiring manager unfamiliar with the codebase reads the README and can repeat back the AI design principles in their own words.
- **As the developer presenting**, I have a 3-minute demo script that walks through manual booking → AI booking → policy question → handoff → trace dashboard in one breath.
  - *Acceptance:* the script is in the repo; a dry-run lands at 3:00 ± 0:15.
- **As an architect**, I can read `docs/architecture.md` and see the request flow, data flow, and channel topology on one page.
  - *Acceptance:* a Mermaid diagram covers HTTP, SSE, agent loop, DB, RAG, Stripe webhook, and channel adapters. No prose without a diagram counterpart.

## Functional Requirements

1. **`README.md` rewrite** — new top-level sections: *Why I Built This*, *What It Demonstrates*, *Architecture* (with diagram + link to architecture.md), *Demo Flow*, *Key AI Design Principles*, *Running It*, *Tests*. The "Why I Built This" section explicitly frames PlayDesk as a COSReady-adjacent vertical demo.
2. **`docs/architecture.md`** — Mermaid diagrams for: request flow, agent loop, channel adapters, RAG pipeline, Stripe webhook. One page, dense, no `TODO`s.
3. **`docs/ai-design-decisions.md`** — captures: database-as-source-of-truth, LLM-suggests-backend-validates, RAG vs SQL partition, human handoff design, prompt versioning, channel adapter abstraction, evals + quality monitoring. Each decision has a one-paragraph rationale and a "what would change v2.1" line.
4. **`docs/demo-script.md`** — both 3-minute and 10-minute versions, with cue cards: what to click, what to say, what to watch the trace dashboard show. Same script supports a Loom recording later.
5. **`screenshots/`** — captures (high-DPI PNG): booking flow steps 1–4, confirmation, chat with streaming tokens + tool chip + booking card, admin dashboard, trace dashboard, handoff queue, bilingual side-by-side. Named `01-booking-step-1.png` etc.
6. **`docs/reports/ai-quality-report.md`** — generated by Stream C, committed as a current snapshot here.

## Non-Functional Requirements

- All docs are reviewer-ready: no placeholders, no broken internal links, no `TODO` markers.
- Mermaid diagrams render on github.com without a plugin.
- Screenshots are <500KB each (PNG-optimised).
- The 3-minute demo script must fit on one printed page.

## Success Criteria

- A reviewer who has never seen the repo can articulate the project's "why" and "how" after 5 minutes with the README.
- The demo script timed against the live stack lands at 3:00 ± 0:15.
- `docs/architecture.md` covers every cross-component data flow shipped through v2.
- The interview package is open-able from a fresh `git clone` with no missing files.

## Constraints & Assumptions

- Depends on streams A, B, C, D being merged to main — the architecture diagram and screenshots must reflect shipped state.
- Existing `CLAUDE.md` stays as the agent-collaboration doc; this stream's `README.md` is the reviewer-facing surface.

## Out of Scope

- A separate "marketing" README.
- Video editing (Loom recording is a follow-up).

## Dependencies

- **Hard:** streams `ai-traces`, `operations`, `quality-metrics`, `channels` must be merged before screenshots are taken.
- Reuses `docs/voice-ai-readiness.md` from Stream D and `docs/reports/ai-quality-report.md` from Stream C.
